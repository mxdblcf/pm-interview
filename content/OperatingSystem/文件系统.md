## 一、总体架构：从用户态到磁盘

Linux 文件系统采用**分层架构**，从用户态的 `open()/read()/write()` 系统调用，经过内核多层抽象，最终落盘到物理存储设备。

## 二、VFS 虚拟文件系统 —— 内核的统一抽象层

### 2.1 VFS 的核心作用

VFS 是 Linux 文件系统的**灵魂**。它定义了一套**统一的接口**，使得上层应用无需关心底层是 ext4、XFS 还是网络文件系统（NFS），都可以使用相同的系统调用操作文件。

### 2.2 VFS 四大核心对象

```
┌─────────────────────────────────────────────────────────────────┐
│                        VFS 四大核心对象                           │
│                                                                 │
│  ┌─────────────────────┐         ┌─────────────────────┐        │
│  │   superblock         │  1 : N  │     inode            │        │
│  │   (超级块)            │ ──────→ │     (索引节点)        │        │
│  │─────────────────────│         │─────────────────────│        │
│  │ s_type              │         │ i_ino   (编号)       │        │
│  │ s_blocksize         │         │ i_mode  (权限)       │        │
│  │ s_op (操作函数表)     │         │ i_uid   (属主)       │        │
│  │ s_root (根dentry)    │         │ i_size  (大小)       │        │
│  │ s_inodes (inode链表) │         │ i_blocks(块数)       │        │
│  │                     │         │ i_op  (inode操作表)   │        │
│  │ 描述整个文件系统的     │         │ i_fop (文件操作表)    │        │
│  │ 元信息               │         │                     │        │
│  └─────────────────────┘         │ 描述单个文件的元数据   │        │
│                                  └───────┬─────────────┘        │
│                                    1 : N │                      │
│                                          ▼                      │
│  ┌─────────────────────┐         ┌─────────────────────┐        │
│  │   file               │  N : 1  │     dentry           │        │
│  │   (文件对象)          │ ──────→ │     (目录项)          │        │
│  │─────────────────────│         │─────────────────────│        │
│  │ f_path              │         │ d_name  (文件名)     │        │
│  │ f_inode             │         │ d_inode (指向inode)  │        │
│  │ f_pos (偏移量)       │         │ d_parent(父目录)     │        │
│  │ f_op (文件操作表)     │         │ d_subdirs(子目录)    │        │
│  │ f_flags             │         │ d_op (操作函数表)     │        │
│  │                     │         │                     │        │
│  │ 进程打开文件的实例     │         │ 文件名到inode的       │        │
│  │                     │         │ 映射缓存             │        │
│  └─────────────────────┘         └─────────────────────┘        │
│                                                                 │
│  关系图:                                                         │
│  superblock ──1:N──→ inode ←──N:1── dentry ←──N:1── file        │
│                        ▲                                        │
│                        └── file 也直接指向 inode                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 四大对象详解

| 对象 | 内核结构体 | 生命周期 | 核心职责 |
|------|-----------|---------|---------|
| **超级块 (superblock)** | `struct super_block` | 文件系统挂载时创建 | 描述文件系统整体信息（块大小、inode 总数、空闲块数等） |
| **索引节点 (inode)** | `struct inode` | 文件首次访问时从磁盘读入 | 描述文件元数据（权限、大小、时间戳、数据块位置） |
| **目录项 (dentry)** | `struct dentry` | 路径查找时动态创建并缓存 | 将文件名映射到 inode，加速路径查找 |
| **文件对象 (file)** | `struct file` | `open()` 时创建，`close()` 时销毁 | 表示进程打开的一个文件实例（偏移量、访问模式） |

> **面试高频题**：*inode 和 dentry 的区别？*
> - inode 存在于磁盘上（持久化），描述文件**本身**的属性
> - dentry 仅存在于内存中（缓存），描述文件**名字**到 inode 的映射关系
> - 一个 inode 可以对应多个 dentry（硬链接）

## 三、文件描述符与进程文件表

### 3.1 从 fd 到磁盘的查找链

```
  进程 (task_struct)
       │
       │ files 指针
       ▼
  files_struct
       │
       │ fdt 指针
       ▼
  fdtable (文件描述符表)
  ┌────┬──────────────┐
  │ fd │   指向        │
  ├────┼──────────────┤        ┌─── 系统级 ───┐        ┌─── 磁盘 ───┐
  │  0 │ ──→ stdin    │        │              │        │            │
  │  1 │ ──→ stdout   │        │              │        │            │
  │  2 │ ──→ stderr   │        │              │        │            │
  │  3 │ ──────────────────→ struct file      │        │            │
  │  4 │ ──→ ...      │     │  f_pos (偏移)   │        │            │
  └────┴──────────────┘     │  f_flags       │        │            │
                            │  f_path.dentry ─────→ struct dentry  │
                            │                │     │  d_name      │
                            └────────────────┘     │  d_inode ────────→ struct inode
                                                   │              │     │  i_ino
                                                   └──────────────┘     │  i_size
                                                                        │  数据块指针 ──→ 磁盘数据块
                                                                        └────────────┘
```

### 3.2 fork / dup 与 file 对象的关系

```
fork() 场景：父子进程共享 file 对象

  父进程                           子进程
  ┌────────────┐                  ┌────────────┐
  │ fd 3 ──────┼──────┐     ┌────┼── fd 3     │
  └────────────┘      │     │    └────────────┘
                      ▼     ▼
                 ┌──────────────┐
                 │  struct file  │  ← 共享同一个（共享 f_pos）
                 │  f_pos = 100  │
                 └──────┬───────┘
                        ▼
                   struct dentry ──→ struct inode

dup() 场景：同一进程内两个 fd 指向同一 file 对象

  进程
  ┌────────────┐
  │ fd 3 ──────┼──────┐
  │ fd 7 ──────┼──────┤  ← dup2(3, 7) 两个 fd 共享同一 file
  └────────────┘      │
                      ▼
                 ┌──────────────┐
                 │  struct file  │
                 └──────────────┘
```

> **面试关键点**：
> - `fd` 是**进程级别**的，不同进程的 fd=3 可能指向不同文件
> - `struct file` 是**系统级别**的，`fork()` 后父子进程共享同一个 file 对象
> - `dup()/dup2()` 创建的新 fd 也指向同一个 file 对象（共享偏移量）

## 四、磁盘布局 —— 以 ext4 为例

### 4.1 ext4 磁盘整体布局

```
 ┌────────────┬────────────┬────────────┬────────────┬─────┬────────────┐
 │  引导块     │   块组 0    │   块组 1    │   块组 2    │ ... │   块组 N    │
 │ Boot Block │ Block Grp 0│ Block Grp 1│ Block Grp 2│     │ Block Grp N│
 │  1024 B    │            │            │            │     │            │
 └────────────┴─────┬──────┴────────────┴────────────┴─────┴────────────┘
                    │
                    ▼  块组内部结构:
 ┌──────────┬──────────┬──────────┬──────────┬──────────┬─────────────────┐
 │  超级块   │  组描述符  │ 数据块位图 │ inode位图 │ inode表  │   数据块区域      │
 │  Super   │   GDT    │  Block   │  inode   │  inode   │   Data Blocks   │
 │  Block   │          │  Bitmap  │  Bitmap  │  Table   │                 │
 │  1 block │ N blocks │ 1 block  │ 1 block  │ N blocks │   剩余空间       │
 └──────────┴──────────┴──────────┴──────────┴──────────┴─────────────────┘
```

### 4.2 各部分职责

| 组件 | 大小 | 职责 |
|------|------|------|
| **超级块** | 1 block | 文件系统全局元信息：魔数、块大小、inode/块总数、挂载次数等 |
| **组描述符表 (GDT)** | N blocks | 描述每个块组的位图位置、inode 表位置、空闲计数等 |
| **块位图** | 1 block | 标记块组内每个数据块的使用状态（0=空闲, 1=占用） |
| **inode 位图** | 1 block | 标记块组内每个 inode 的使用状态 |
| **inode 表** | N blocks | 存储该块组内所有 inode（默认每个 inode 256 字节） |
| **数据块** | 剩余空间 | 存储实际文件内容和目录项 |

## 五、inode 与数据块寻址

### 5.1 传统 ext2/ext3 多级索引

```
                    inode 结构
              ┌─────────────────┐
              │  i_mode         │
              │  i_uid          │
              │  i_size         │
              │  i_atime        │
              │  i_mtime        │
              │  i_ctime        │
              │  i_blocks       │
              │─────────────────│
              │  直接指针 0  ────┼──────→ [数据块 0]
              │  直接指针 1  ────┼──────→ [数据块 1]
              │  直接指针 2  ────┼──────→ [数据块 2]
              │  ...            │         ...
              │  直接指针 11 ────┼──────→ [数据块 11]       共 12 个，48KB
              │─────────────────│
              │  一级间接 ───────┼──→ ┌──────────┐
              │                 │    │ 指针 → [块12] │
              │                 │    │ 指针 → [块13] │     1024 个指针
              │                 │    │ ...          │     → 4MB
              │                 │    │ 指针 → [块1035]│
              │                 │    └──────────┘
              │─────────────────│
              │  二级间接 ───────┼──→ ┌──────────┐
              │                 │    │ 指针 ──→ [间接块] ──→ [数据块...]  │
              │                 │    │ 指针 ──→ [间接块] ──→ [数据块...]  │
              │                 │    │ ...                              │
              │                 │    └──────────┘     1024² → 4GB
              │─────────────────│
              │  三级间接 ───────┼──→ ┌──────────┐
              │                 │    │ 指针 ──→ [二级间接块]             │
              │                 │    │           └→ [间接块]→[数据块]    │
              │                 │    │ ...                              │
              │                 │    └──────────┘     1024³ → 4TB
              └─────────────────┘
```

**寻址能力计算（块大小 4KB, 指针 4 字节）**：

| 层级 | 可寻址块数 | 最大文件大小 |
|------|-----------|------------|
| 直接指针 (12个) | 12 | 48 KB |
| 一级间接 | 1024 | 4 MB |
| 二级间接 | 1024² = 1M | 4 GB |
| 三级间接 | 1024³ = 1G | 4 TB |
| **合计** | **约 1G + 1M + 1036** | **约 4 TB** |

### 5.2 ext4 Extent Tree（现代方案）

ext4 引入 **Extent** 取代传统间接块，大幅减少元数据开销：

```
  一个 Extent 描述一段【连续】的物理块:

  struct ext4_extent {
      __le32  ee_block;    // 逻辑起始块号
      __le16  ee_len;      // 连续块数量 (最大 32768 = 128MB)
      __le16  ee_start_hi; // 物理起始块号高16位
      __le32  ee_start_lo; // 物理起始块号低32位
  };

  示例: 一个 600 块的文件 → 只需 2 个 Extent

  inode (i_block 60字节空间)
  ┌─────────────────────────┐
  │ Extent Header            │
  │   eh_entries = 2         │
  ├─────────────────────────┤
  │ Extent 1                 │         磁盘
  │   逻辑块 0~99            │ ──→  ┌──────────────────────────┐
  │   物理块 1000~1099       │      │ 物理块 1000-1099          │
  │   连续 100 个块 (400KB)   │      │ 400KB 连续数据           │
  ├─────────────────────────┤      └──────────────────────────┘
  │ Extent 2                 │
  │   逻辑块 100~599         │ ──→  ┌──────────────────────────┐
  │   物理块 5000~5499       │      │ 物理块 5000-5499          │
  │   连续 500 个块 (2MB)     │      │ 2MB 连续数据             │
  └─────────────────────────┘      └──────────────────────────┘

  当 Extent 数量超过 4 个 → 自动扩展为 Extent Tree (B+树)
```

> **面试高频题**：*ext4 相比 ext3 的改进？*
> - Extent 替代间接块 → 减少大文件元数据开销，提升顺序读写性能
> - 延迟分配 (Delayed Allocation) → 减少碎片
> - 多块分配 (Multiblock Allocation) → 一次分配多个连续块
> - 日志校验和 → 提高可靠性
> - 支持最大 1 EB 文件系统、16 TB 单文件

## 六、Page Cache 页缓存 —— 性能的关键

### 6.1 读操作路径

```
  App                VFS              Page Cache          ext4            块层           磁盘
   │                  │                   │                │               │              │
   │  read(fd,buf,n)  │                   │                │               │              │
   │ ────────────────→│                   │                │               │              │
   │                  │  查找页缓存        │                │               │              │
   │                  │ ─────────────────→│                │               │              │
   │                  │                   │                │               │              │
   │                  │      ┌── 缓存命中 ─┤                │               │              │
   │                  │      │            │                │               │              │
   │  ← 拷贝到buf ────│← ────┘            │                │               │              │
   │                  │                   │                │               │              │
   │                  │      ┌── 缓存未命中┤                │               │              │
   │                  │      │            │  请求读取       │               │              │
   │                  │      │            │ ──────────────→│  提交BIO       │              │
   │                  │      │            │                │ ─────────────→│  DMA读磁盘    │
   │                  │      │            │                │               │ ────────────→│
   │                  │      │            │                │               │   数据返回    │
   │                  │      │            │                │  ←────────────│←────────────│
   │                  │      │            │  ←─────────────│               │              │
   │                  │      │   数据填入缓存←─────────────│               │              │
   │  ← 拷贝到buf ────│← ────┘            │                │               │              │
   │                  │                   │                │               │              │
```

### 6.2 写操作路径

```
  App                VFS              Page Cache                        磁盘
   │                  │                   │                              │
   │  write(fd,buf,n) │                   │                              │
   │ ────────────────→│                   │                              │
   │                  │  写入页缓存         │                              │
   │                  │  (标记 Dirty 脏页)  │                              │
   │                  │ ─────────────────→│                              │
   │                  │  ← 完成           │                              │
   │  ← 返回(即完成!) │                   │                              │
   │                  │                   │                              │
   │           ★ 注意: write() 返回时数据还没到磁盘！                       │
   │                  │                   │                              │
   │                  │           后台回写线程 (pdflush/writeback)          │
   │                  │                   │  ──── 异步刷脏页 ────────────→│
   │                  │                   │                              │
```

### 6.3 Page Cache 关键机制

| 机制 | 说明 |
|------|------|
| **Radix Tree / XArray** | 按文件偏移量快速索引缓存页（O(log n) 查找） |
| **预读 (Readahead)** | 检测到顺序读模式时，提前读取后续页 → 默认预读窗口 128KB |
| **脏页回写** | `dirty_ratio`（默认 20%）：脏页占内存比超阈值时**同步**写回 |
| **后台回写** | `dirty_background_ratio`（默认 10%）：超阈值时**异步**后台回写 |
| **内存回收** | 内存紧张时 LRU 算法回收**干净页**，脏页先回写再回收 |

> **面试关键点**：
> - `write()` 返回**不代表**数据已写入磁盘，仅写入 Page Cache
> - 要保证数据持久化，需调用 `fsync()` / `fdatasync()` / `O_SYNC`
> - 掉电丢数据的根因就是 Page Cache 中的脏页还未回写

  │  将请求转换为设备命令           │
  └──────────┬───────────────────┘
             │
             ▼
  ┌──────────────────────────────┐
  │  硬件控制器 → DMA 传输         │
  └──────────┬───────────────────┘
             │
             ▼
  ┌──────────────────────────────┐
  │  物理介质                     │
  │  HDD磁盘 / SSD NAND Flash    │
  └──────────────────────────────┘
```

### 7.2 BIO 结构

```c
struct bio {
    struct block_device *bi_bdev;    // 目标块设备
    sector_t            bi_sector;   // 起始扇区号
    unsigned int        bi_size;     // 数据大小
    struct bio_vec     *bi_io_vec;   // 内存页数组 (page, offset, len)
    bio_end_io_t       *bi_end_io;   // I/O 完成回调
    // ...
};
```

### 7.3 I/O 调度器对比

| 调度器 | 适用场景 | 算法思想 |
|--------|---------|---------|
| **mq-deadline** | HDD（默认） | 设置读写请求的截止时间，防饥饿，兼顾公平与吞吐 |
| **BFQ** | 桌面/交互场景 | Budget Fair Queueing，保证低延迟和公平性 |
| **kyber** | 高速 SSD | 双队列（同步/异步），针对低延迟设备优化 |
| **none (noop)** | NVMe | 不做排序/合并，设备自身有调度能力 |

> **面试关键点**：NVMe SSD 通常使用 `none` 调度器，因为设备内部已有高效调度，内核层面再排序反而增加延迟。

## 八、日志文件系统 (Journaling) —— 崩溃一致性

### 8.1 日志写操作流程

```
  ┌──────────┐     ┌──────────┐     ┌──────────────────┐     ┌──────────────┐
  │ 1.写日志  │────→│ 2.提交日志│────→│ 3.写数据到目标位置 │────→│ 4.释放日志空间 │
  │ Journal  │     │ Commit   │     │ Checkpoint        │     │              │
  └──────────┘     └──────────┘     └──────────────────┘     └──────────────┘
```

### 8.2 三种日志模式对比

```
  ┌──────────────────────┬──────────────────────┬──────────────────────┐
  │    Journal 模式       │   Ordered 模式 ★      │   Writeback 模式     │
  │                      │    (ext4 默认)        │                      │
  ├──────────────────────┼──────────────────────┼──────────────────────┤
  │                      │                      │                      │
  │  元数据 + 数据        │  仅元数据写日志        │   仅元数据写日志       │
  │  都写日志             │  数据先于元数据落盘     │   数据写入顺序不保证   │
  │                      │                      │                      │
  │  ─────────────────   │  ─────────────────   │   ─────────────────  │
  │  ★ 最安全            │  ★ 默认模式           │   ★ 性能最好          │
  │  ★ 性能最差          │  ★ 安全与性能平衡      │   ★ 可能数据不一致     │
  │                      │                      │                      │
  └──────────────────────┴──────────────────────┴──────────────────────┘
```

### 8.3 崩溃恢复原理

```
  正常流程:
  TxBegin → [日志写入] → TxEnd(Commit) → [检查点写入] → 释放日志

  ─────────────────────────────────────────────────────────────────

  崩溃场景 1:
  TxBegin → [日志部分写入] → 💥 崩溃
  恢复策略:  丢弃不完整事务 → 文件系统不受影响 ✓

  ─────────────────────────────────────────────────────────────────

  崩溃场景 2:
  TxBegin → [日志写入] → TxEnd → [检查点部分写入] → 💥 崩溃
  恢复策略:  重放日志(Replay) → 将日志中的完整事务重新写入目标位置 ✓
```

> **面试高频题**：*ext4 默认的 ordered 模式如何保证一致性？*
> 1. 先将**数据块**写入磁盘目标位置
> 2. 再将**元数据变更**写入日志区域
> 3. 提交日志事务
> 4. 将元数据写入磁盘目标位置
> 5. 如果在步骤 1 崩溃 → 元数据未更新，旧数据仍然一致
> 6. 如果在步骤 2-3 崩溃 → 日志不完整，丢弃事务
> 7. 如果在步骤 4 崩溃 → 重放日志恢复元数据

## 十、硬链接 vs 软链接（符号链接）

```
  ═══════════════════ 硬链接 (Hard Link) ═══════════════════

  file_a (dentry) ──→ ┌──────────────────┐
                      │   inode 12345     │
  file_b (dentry) ──→ │   i_nlink = 2    │ ──→ [数据块]
                      │   数据块指针      │
                      └──────────────────┘
      两个文件名指向同一个 inode，删一个不影响另一个

  ═══════════════════ 软链接 (Symbolic Link) ═══════════════

  link_c (dentry) ──→ ┌──────────────────┐
                      │   inode 99999     │
                      │   类型: 符号链接   │
                      │   内容: "/path/   │
                      │         to/target"│     路径解析
                      └──────────────────┘ ·····▶  target (dentry)
                                                      │
                                               ┌──────┴──────────┐
                                               │   inode 12345    │
                                               │   数据块指针      │──→ [数据块]
                                               └─────────────────┘
```

| 特性 | 硬链接 | 软链接（符号链接） |
|------|--------|-----------------|
| 本质 | 新 dentry 指向**同一 inode** | 新 inode，内容是**目标路径字符串** |
| 跨文件系统 | ❌ 不可以 | ✅ 可以 |
| 链接目录 | ❌ 不可以（防环） | ✅ 可以 |
| 目标删除后 | 仍可访问（inode 未释放） | 悬挂链接（dangling link） |
| inode 数 | 不增加 | +1（链接本身的 inode） |
| `i_nlink` | +1 | 不变（目标的） |

## 十四、mmap 内存映射文件

### 14.1 mmap 原理

```
  进程虚拟地址空间                    内核                         磁盘
  ┌──────────────┐
  │  代码段 .text │
  ├──────────────┤
  │  数据段 .data │
  ├──────────────┤
  │  堆 heap  ↓  │
  │              │
  │              │
  │  ............│
  │              │
  │  mmap 映射区 │──────→ ┌──────────────────┐
  │  0x7f..000   │        │  vm_area_struct   │
  │  ~           │        │  vm_start/vm_end  │
  │  0x7f..fff   │        │  vm_file          │
  │              │        │  vm_pgoff         │
  │  ............│        └────────┬─────────┘
  │              │                 │
  │  栈 stack ↑  │                 │ page fault 时
  └──────────────┘                 │ 按需加载
                                   ▼
                           ┌──────────────────┐        ┌──────────┐
                           │   Page Cache      │ ←────→ │ 文件数据块│
                           │   页缓存          │  缓存   │          │
                           └──────────────────┘  未命中  └──────────┘
                                   │             时读盘
                                   │
                           ★ 关键: mmap 直接将 Page Cache
                             的物理页映射到用户虚拟地址
                             → 零拷贝!
```

### 14.2 mmap vs read/write

| 对比维度 | mmap | read/write |
|---------|------|-----------|
| **数据拷贝** | 零拷贝（直接访问 Page Cache 页） | 需从 Page Cache 拷贝到用户 buffer |
| **系统调用开销** | 首次 mmap 一次，之后无系统调用 | 每次读写都需系统调用 |
| **适用场景** | 大文件随机访问、共享内存 | 小文件、顺序访问、流式处理 |
| **缺点** | TLB 压力大、page fault 开销 | 多次内存拷贝 |

> **面试关键点**：
> - `MAP_SHARED` ：多进程可见，修改会写回文件
> - `MAP_PRIVATE` ：COW（写时复制），修改不影响文件和其他进程
> - 数据库（如 MongoDB 早期版本）大量使用 mmap 映射数据文件

## 十五、Direct I/O vs Buffered I/O

```
  ════════ Buffered I/O（默认） ════════

  用户 Buffer ──①数据拷贝──→ Page Cache ──②异步回写──→ 磁盘

  ════════ Direct I/O (O_DIRECT) ════════

  用户 Buffer ──①直接 DMA──────────────────────────→ 磁盘
  (需对齐)          绕过 Page Cache!
```

| 特性 | Buffered I/O | Direct I/O |
|------|-------------|-----------|
| 经过 Page Cache | ✅ 是 | ❌ 否，绕过 |
| 内存对齐要求 | 无 | 用户 buffer 必须按扇区/页对齐 |
| 适用场景 | 通用场景 | 数据库（自己管理缓存）、大文件流式处理 |
| 代表用户 | 大部分应用 | MySQL InnoDB（`innodb_flush_method=O_DIRECT`）、PostgreSQL |

> **面试关键点**：数据库使用 Direct I/O 的原因是它们有自己的 Buffer Pool，Page Cache 反而造成**双重缓存**，浪费内存。

## 十六、面试高频问题速查

### Q1: 一个文件的读写过程是怎样的？
**答**：`read()` → 系统调用 → VFS → 查 Page Cache → 命中直接返回 / 未命中则通过文件系统发起块 I/O → 数据填入 Page Cache → 拷贝到用户态。`write()` 类似但写入 Page Cache 即返回，后台异步回写。

### Q2: 为什么 cp 大文件时内存会飙升？
**答**：cp 使用 buffered I/O，读写的数据会经过 Page Cache，大文件的数据都会在 Page Cache 中占用内存。当内存压力大时，内核会通过 LRU 回收这些缓存页。可用 `posix_fadvise(FADV_DONTNEED)` 或 `O_DIRECT` 绕过。

### Q3: 删除文件但磁盘空间不释放？
**答**：`rm` 只是 `unlink()`，inode 的 `i_nlink` 减 1。如果还有进程持有打开的 fd（`i_count > 0`），数据块不会立即释放，直到所有 fd 关闭。用 `lsof | grep deleted` 查看。

### Q4: 软链接和硬链接的底层区别？
**答**：硬链接是新建一个 dentry 指向同一个 inode（`i_nlink++`）；软链接是创建新 inode，其数据内容是目标路径字符串，访问时触发路径重新解析。

### Q5: ext4 的 ordered 日志模式如何保证一致性？
**答**：数据先于元数据落盘。先写数据到目标位置 → 再写元数据到日志 → 提交日志 → 写元数据到目标位置。任何阶段崩溃，要么数据+元数据一致，要么通过日志重放恢复元数据。

### Q6: 什么是 inode 耗尽？
**答**：文件系统创建时 inode 数量固定（`mkfs` 时确定）。如果创建大量小文件，可能出现磁盘空间剩余但 inode 耗尽、无法创建新文件的情况。`df -i` 可查看 inode 使用情况。

### Q7: mmap 和 read/write 的本质区别？
**答**：`read()/write()` 需要**内核态↔用户态数据拷贝**；`mmap()` 将 Page Cache 的物理页直接映射到用户态虚拟地址空间，访问时通过 page fault 按需加载，省去了数据拷贝。代价是 TLB miss 和 page fault 的开销。

# XFS 查看
$ xfs_info /dev/sda1
# 输出中 imaxpct 表示 inode 最大占磁盘空间的百分比
```

### 18.2 inode 数量是怎么确定的？

```
创建文件系统时（mkfs），根据以下公式分配 inode 数量：

  inode 数量 ≈ 磁盘总容量 / bytes-per-inode

  默认 bytes-per-inode = 16384 (即每 16KB 空间分配一个 inode)

  示例：
  ┌─────────────────────────────────────────────────────┐
  │  100 GB 磁盘，bytes-per-inode = 16384               │
  │  inode 数量 = 100GB / 16KB ≈ 6,553,600 个           │
  │  → 最多可创建约 655 万个文件                          │
  └─────────────────────────────────────────────────────┘

  如果存大量小文件（如邮件服务器），可能 inode 耗尽但磁盘空间还有剩余！
```

### 18.3 查看单个文件的最大大小

| 文件系统 | 最大文件大小 | 最大卷大小 | 说明 |
|----------|------------|-----------|------|
| **ext4** | 16 TB | 1 EB | 默认 4K block size；1K block → 最大 16GB |
| **XFS** | 8 EB | 8 EB | 适合超大文件场景 |
| **Btrfs** | 16 EB | 16 EB | COW 文件系统 |
| **NTFS** | 16 TB | 256 TB | Windows 默认 |
| **FAT32** | 4 GB | 2 TB | U 盘常用，注意 4GB 限制！ |

```bash
# ext4 查看块大小（决定了最大文件/卷大小）
$ sudo tune2fs -l /dev/sda1 | grep "Block size"
Block size:               4096

# 块大小与 ext4 最大文件大小的关系：
# 1K block → 最大文件 16 GB
# 2K block → 最大文件 256 GB
# 4K block → 最大文件 16 TB （最常用）
```

### 18.4 如何调节这些限制

#### ① 创建时指定（mkfs 参数）

```bash
# ext4: 创建时指定更多 inode（适合大量小文件场景）
$ mkfs.ext4 -i 4096 /dev/sda1
#         ↑ 每 4KB 分配一个 inode（默认 16KB）
#         → inode 数量变为默认的 4 倍

# ext4: 指定 inode 大小
$ mkfs.ext4 -I 512 /dev/sda1
#         ↑ 每个 inode 占 512 字节（默认 256）
#         带来更多扩展属性空间，但总 inode 数量减少

# ext4: 指定块大小
$ mkfs.ext4 -b 4096 /dev/sda1
#         ↑ 4096 字节的块大小（影响最大文件大小）

# XFS: 创建时指定 inode 比例
$ mkfs.xfs -i maxpct=25 /dev/sda1
#         ↑ 允许最多 25% 的磁盘空间用于 inode（默认 5%）
```

#### ② 创建后调整（有限能力）

```bash
# ext4: 修改 inode 预留空间（在线调整，部分参数）
$ sudo tune2fs -m 3 /dev/sda1
#         ↑ 降低 root 预留空间到 3%（默认 5%），释放更多可用空间

# XFS: 动态扩展 inode
# XFS 的 inode 是动态分配的，不像 ext4 在创建时固定
# 只要磁盘有空间，XFS 可以自动分配更多 inode
$ xfs_growfs /dev/sda1     # 在线扩容

# ⚠️ ext4 的 inode 数量创建后不可增加！
# 唯一的解决办法是备份数据 → 重新 mkfs → 恢复数据
```

#### ③ 实战场景与经验

```
场景 1：邮件服务器（大量小文件）
  → mkfs.ext4 -i 2048     # 每 2KB 分配一个 inode
  → 或改用 XFS（动态 inode）

场景 2：视频存储（少量大文件）
  → mkfs.ext4 -i 65536    # 每 64KB 分配一个 inode（节省 inode 表空间）
  → 或改用 XFS（天然适合大文件）

场景 3：容器环境（大量 overlay 层）
  → 注意 inode 监控：df -i
  → Docker 的 overlay2 下每个文件都消耗底层 inode
```

> **面试高频题**：*如何排查 "No space left on device" 但 df 显示还有空间？*
> 1. 检查 inode：`df -i` — 如果 IUse% = 100%，说明 inode 耗尽
> 2. 检查是否有被删除但仍被进程持有的文件：`lsof +L1`
> 3. 检查保留空间：ext4 默认为 root 预留 5%，普通用户看不到这部分空间
>    - 调整：`tune2fs -m 1 /dev/sda1`（改为 1%）
