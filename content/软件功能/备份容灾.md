# 备份容灾硬核面试手册

> 高密度技术版 · VMware / PostgreSQL / 重删底层 / 勒索恢复 / 边界计算

---

## 1. 本质与核心矛盾

备份解决「回到过去」，容灾解决「业务别停」。工程核心是：在数据高频变化的生产系统上，以可控的 RPO / RTO 构建可隔离、可追溯、可验证的时间机器。

> ⚡ 没有恢复演练 + 不可变副本的备份体系，不具备真正的生产级容灾能力。

| 概念 | 核心目标 | 防逻辑错误 | 历史深度 | 快速接管 |
|------|---------|-----------|---------|---------|
| Backup 备份 | 数据可恢复到历史点 | ✓ 强 | 深，取决于保留策略 | 不一定 |
| Archive 归档 | 长期合规留存 | 一般 | 极深（年级） | ✗ |
| Replication 复制 | 近实时副本 | ✗ 弱（错误同步） | 浅（小时级） | ✓ 快 |
| DR 容灾 | 整体业务恢复 | 视是否结合备份 | 视方案 | ✓ |
| HA 高可用 | 单点故障不中断 | ✗ | 无 | ✓ 自动 |

> ⚠️ 复制不等于备份：误删、勒索加密、应用 Bug 写坏数据，都会被同步到副本端。历史恢复点 + 副本隔离才是根本。

## 2. VMware 备份原理（深度版）

### 2.1 整体数据流

VMware 备份不读宿主机文件系统，而是通过 **vSphere VADP**（vStorage APIs for Data Protection）走 ESXi 内核层拿数据。

```
Backup Server
  │
  ├─① 调用 vCenter/ESXi VADP API → 创建 VM Snapshot
  │     └─ Snapshot = 新建 delta VMDK（-delta.vmdk），原 VMDK 冻结
  │
  ├─② 查询 CBT（Changed Block Tracking）
  │     └─ 返回自上次备份以来变化的 <offset, length> 列表
  │
  ├─③ Proxy 读取变化块（三种模式）
  │     ├─ HotAdd：Proxy VM 热挂载源 VMDK 到自身，走 SCSI 本地读
  │     ├─ SAN：Proxy 直接通过 FC/iSCSI 读存储 LUN（需访问权限）
  │     └─ NBD/NBDSSL：走 TCP 管理网络，ESXi 作 NFC server 提供块流
  │
  ├─④ 数据处理（压缩/加密/重删）→ 写入 Repository
  │
  └─⑤ 删除 Snapshot → ESXi 触发 Snapshot Consolidation
```

### 2.2 CBT（Changed Block Tracking）底层机制

CBT 是 ESXi kernel 在 VMDK I/O 路径上维护的一个 **bitmap 结构**，每个 bit 对应一个扇区区间。

- CBT 由 VMDK 元数据文件（`-ctk.vmdk`）和内存中的位图共同维护
- 每次写操作进入 ESXi storage stack 时，对应扇区 bit 被置 1
- 备份完成后，备份软件调用 `ResetChangedBlockTracking` 清零（或传入 changeID 获取增量）
- CBT 使用 **changeID**（字符串 token）而非时间戳标识版本，保证跨 vMotion 可追溯
- `QueryChangedDiskAreas(changeId)` 返回变化 extent 列表，**粒度 512KB 对齐**

> ⚠️ CBT 已知失效场景：(1) VM 重新配置（磁盘扩容/迁移）；(2) snapshot 链异常；(3) ESXi host 崩溃重启；(4) stun/unstun 周期过长导致 bitmap 损坏。失效后必须全量扫描，代价等同于首次全备。

### 2.3 三种传输模式对比

| 模式 | 数据路径 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **HotAdd** | Proxy VM → SCSI 挂载源 VMDK → 本地读 | 带宽高，无需额外网络 | Proxy 需在 ESXi 上运行；挂载点数量限制；snapshot consolidate 需所有挂载释放 | 虚拟化主流方案 |
| **SAN Mode** | Proxy → FC/iSCSI → 存储阵列 LUN | 最高吞吐，0 ESXi CPU | 需 SAN Zoning/Masking；LUN 只读挂载复杂；仅支持块存储 | 大型 SAN 环境 |
| **NBD/NBDSSL** | Proxy → TCP → ESXi NFC Server | 无需存储访问权限 | ESXi CPU 开销高；带宽受管理网络限制 | 小环境/无 SAN |

### 2.4 快照（Snapshot）生命周期——最常踩的坑

ESXi 快照本质是 **redo-log 链**，写操作重定向到 delta VMDK，读操作可能跨多层链：

```
base.vmdk（原始磁盘，快照后变为只读）
  └─ base-delta.vmdk        ← Snapshot 1（所有写入进这里）
       └─ base-delta-000001.vmdk  ← Snapshot 2
            └─ base-delta-000002.vmdk  ← Snapshot 3（当前写入）
```

- 链越深，读 I/O 放大越严重（每次读可能跨多个 delta 文件）
- 删快照时触发 **consolidation**：ESXi 需将 delta 合并回基础盘，过程中有 VM **stun**（暂停写 I/O）
- 大 delta 文件 consolidate 时间可能几分钟到几十分钟，期间 VM 性能显著下降
- 长时间保留快照（超过 24–72h）是生产事故高发原因
- 备份软件的快照应在数据读取完毕后**立即删除**，不得积压

### 2.5 应用一致性 vs 崩溃一致性

| 级别 | 实现方式 | 适用场景 | 数据库恢复性 |
|------|---------|---------|------------|
| Crash-Consistent | 直接快照，不 quiesce | 无状态服务 | 数据库可能有未提交事务，需 journal recovery |
| File-System Consistent | VSS 或 Linux freeze（fsfreeze），flush buffer cache | 有写缓存的文件服务 | FS 层一致，DB 内部不保证 |
| App-Consistent | VSS Writer + DB agent quiesce | SQL Server, Exchange, Oracle | 事务完整，可直接 mount 使用 |

> ⚠️ VMware Tools quiesce ≠ 应用一致。quiesce 只能做到 VSS provider 层，如果数据库没有注册 VSS Writer 或 Writer 报错，仍是文件系统一致级别。必须在备份日志中验证 VSS Writer 状态为 `Stable`。

## 3. 重删底层原理（深度版）

### 3.1 分块策略

重删的第一步是把数据流切成块（Chunking），切块策略直接决定重删效率：

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **Fixed-Size Chunking** | 按固定大小（如 4KB/8KB）切块 | 简单，索引规模可预测 | 文件插入/删除后块边界整体偏移，重删率骤降 |
| **CDC（Content-Defined Chunking）** | 用 Rabin Fingerprint 滚动窗口，遇到特定 hash 模式就切 | 块边界与内容相关，抗偏移能力强 | CPU 开销较大；块大小不均匀（需设 min/max 限制） |
| **Variable + Anchored** | CDC 基础上加锚点对齐，限定 min 512B / max 8KB | 重删率和索引规模平衡 | 实现复杂度高 |

**Rabin Fingerprint 滚动窗口原理：**

```
滑动窗口（如 48 字节）在字节流上滚动，
每移动一个字节，用多项式滚动 hash 更新指纹值：

  hash = (hash × base + new_byte - old_byte × base^w) mod prime

当 hash & mask == pattern 时，触发切块边界。
这样切块点由数据内容决定，而非固定偏移。
```

企业备份产品（Veeam, Commvault, Rubrik）普遍使用 CDC 变长分块，原因是虚拟机磁盘、数据库页面在增量变化时不是固定偏移变化的。

### 3.2 哈希算法选型

| 算法 | 输出长度 | 速度 | 碰撞概率 | 用途 |
|------|---------|------|---------|------|
| MD5 | 128 bit | 快 | 高（已有碰撞攻击） | 不推荐用于新系统 |
| SHA-1 | 160 bit | 中 | 中（理论碰撞已证明） | 旧系统兼容 |
| SHA-256 | 256 bit | 慢 | 极低（2^-128 量级） | 安全要求高场景 |
| **xxHash / BLAKE3** | 64–256 bit | **极快（SIMD 加速）** | 低（非密码学安全，但碰撞极难） | 现代备份产品首选性能指纹 |

大多数现代备份平台用非密码学 hash（xxHash/BLAKE3）做重删指纹，速度是 SHA-256 的 5–10 倍，碰撞概率在实际数据规模下可忽略。但需要额外 integrity check 保证数据正确性。

### 3.3 哈希碰撞处理机制

Hash 碰撞（两个不同数据块算出相同 hash）在理论上存在，生产系统的处理策略由强到弱：

**策略 A：Strong Hash + Byte-by-Byte Compare（最强）**
```
1. 用快速 hash（xxHash）匹配候选块
2. 命中后用 SHA-256 二次确认
3. 仍有疑义时进行字节逐一比对
代价：写入延迟增加，但碰撞误判率接近 0
```

**策略 B：双 Hash 独立校验**
```
同时计算 SHA-1 + xxHash，两者均匹配才认为重复
双 hash 碰撞概率 ≈ 2^-160 × 2^-64 → 极低，工程上可忽略
```

**策略 C：Pure Trust Hash + 后台 Scrubbing**
```
完全信任 hash，依赖后台 scrubbing 任务周期性校验数据块完整性
发现损坏时从其他副本或 parity 恢复
适合海量数据 + 高性能要求场景
```

> 💡 生产级备份系统（Data Domain, Veeam）通常采用策略 B 或 A，并辅以后台 scrubbing 任务。纯信任 hash 在 PB 级备份数据下存在不可接受的概率风险。

### 3.4 重删索引（Dedup Index）内存模型

```
Hash Index 结构（Key-Value 存储）：

  Key:   chunk_hash（32 byte SHA-256）
  Value: {
    container_id,     // 数据块所在 container 文件 ID
    offset,           // container 内偏移
    length,           // 块长度
    ref_count,        // 引用计数（为 0 时 GC）
  }

典型规模：
  1 PB 数据 @ 平均块大小 8KB = 1.25 亿个 chunk
  每条索引约 50B → 索引总大小 ≈ 6 GB
  通常需要全部或热索引驻内存，冷索引用 SSD 缓存
```

> ⚠️ 内存是重删系统的第一瓶颈。索引 cache miss → SSD/HDD 随机读 → 写入延迟暴增。Data Domain 的 DRAM 配置通常以 TB 存储容量 1GB RAM 为基准线。

### 3.5 GC（垃圾回收）机制

重删系统的 GC 是高风险操作：过期恢复点删除后，需要扫描哪些 chunk 引用计数归零，然后回收空间。

- **Mark-and-Sweep 两阶段**：① 标记所有活跃恢复点引用的 chunk；② 清除未被引用的 chunk
- GC 期间 I/O 放大严重，通常在业务低峰窗口执行
- GC 和活跃备份任务争抢仓库资源是**合成全备 + 重删组合的常见性能陷阱**
- 引用计数（ref_count）必须**原子更新**，否则并发写入时会出现幽灵引用

### 3.6 源端重删 vs 目标端重删

| 维度 | 源端重删（Client-Side） | 目标端重删（Target-Side） |
|------|----------------------|------------------------|
| 网络传输量 | 极低（只传非重复块） | 全量传输后仓库端去重 |
| 源端 CPU 开销 | 高（hash 计算、索引查询在客户端） | 低 |
| WAN 复制优化 | 显著降低复制带宽 | 需仓库端 DD Boost / NDMP 协议才能优化 |
| 索引位置 | 分布式，各客户端维护 | 集中在仓库，全局去重效果更强 |
| 实现复杂度 | 高（需客户端 agent） | 低（仓库设备统一处理） |

> Data Domain 的 **DD Boost** 是混合模式：客户端做本地 segment 对比（避免传已有块），仓库端做全局索引——兼顾网络和去重率。

## 4. PostgreSQL 备份原理（深度版）

### 4.1 WAL 机制底层

理解 PG 备份必须先理解 WAL（Write-Ahead Logging）物理结构：

```
WAL 文件：$PGDATA/pg_wal/
命名格式：{timeline}{LSN_high}{LSN_low}
示例：000000010000000100000001
每个 WAL 文件默认 16MB（wal_segment_size）

LSN（Log Sequence Number）：64-bit 单调递增地址
  高32位 = WAL 文件编号
  低32位 = 文件内偏移

WAL Record 结构：
  XLogRecord Header  → resource manager ID, total length, prev LSN
  RmgrData           → 具体操作（HEAP INSERT/UPDATE/DELETE, BTREE 等）
  Block Image（可选）→ Full Page Write（FPW）时包含整页
```

> ⚠️ **Full Page Write（FPW）**：PostgreSQL 在 checkpoint 后第一次修改某页，会把整页写入 WAL。这是为了防止部分页写（Partial Write）导致数据损坏。FPW 会显著增大 WAL 体积（可达 3–5 倍），是 WAL 归档带宽估算必须考虑的因素。

### 4.2 Base Backup + PITR 完整流程

```
步骤 1：pg_basebackup 或 pg_start_backup()
  → PG 强制触发 checkpoint，记录起始 LSN（start_lsn）
  → 进入备份模式（backup_label 写入 $PGDATA）
  → 此后所有 FPW 强制开启（保证物理一致性）

步骤 2：拷贝 $PGDATA 目录（online 热备份）
  → 拷贝过程中数据库正常运行
  → 拷贝的文件可能包含进行中的事务（不一致状态）

步骤 3：pg_stop_backup()
  → 记录结束 LSN（stop_lsn）
  → 生成 backup_label / tablespace_map
  → WAL 持续归档不中断

步骤 4：归档 start_lsn 到 stop_lsn 之间的所有 WAL 段
  → 这些 WAL 是恢复时补齐 base backup 不一致状态的关键

PITR 恢复：
  1. 还原 base backup 到目标目录
  2. 写 recovery.conf / postgresql.conf：
       restore_command = 'cp /archive/%f %p'
       recovery_target_time = '2024-03-15 14:30:00'
  3. 启动 PG → 自动 redo WAL 直到 target time → 暂停等待确认
  4. SELECT pg_wal_replay_resume() → 数据库 promote
```

### 4.3 WAL 归档链断裂——最致命的风险

- `archive_command` 失败（目标不可写、认证失败）→ WAL 积压在 `pg_wal` → 目录撑满磁盘 → PG panic 停库
- `archive_status/ready` 文件积压超过 `wal_keep_size` → 旧 WAL 被覆盖，归档链断裂
- 手动删除 `pg_wal` 中的段文件而不验证是否已归档 → 永久丢失
- 归档目标容量不足导致静默失败 → `archive_command` 返回非 0 → WAL 无限重试但不阻塞写入 → 看起来正常但实际归档早停了
- `archive_timeout` 未设置 → 低负载期间 WAL 段迟迟不归档 → 归档延迟超出 RPO 要求

> ⚠️ 监控 `pg_stat_archiver.last_failed_wal` 和 `archived_count` 的增量是 WAL 归档健康度的核心指标，比监控进程存活更重要。

### 4.4 物理备份 vs 逻辑备份边界

| 维度 | pg_basebackup（物理） | pg_dump（逻辑） |
|------|---------------------|----------------|
| 恢复粒度 | 整个集群 | 单库/单表/单对象 |
| PITR 支持 | ✓ 完整支持 | ✗ 只能恢复到备份时刻 |
| 跨大版本迁移 | ✗ 不支持 | ✓ 主要用途 |
| 大库备份速度 | 快（文件级拷贝） | 慢（SQL 导出） |
| 恢复速度 | 快 | 慢（需重建索引） |
| 部分表恢复 | ✗ 需额外工具 | ✓ `-t tablename` |
| 备份期间锁 | 几乎无（FPW 模式补偿） | SHARE 锁（`--lock-wait-timeout` 控制） |

### 4.5 Streaming Replication 和 WAL 归档的关系

- 流复制（Streaming Replication）是实时 WAL 推送，降低 RPO 到秒级，但**副本也会复制逻辑错误**
- WAL 归档是异步批量归档，提供历史深度，是 PITR 的基础
- 两者不互斥：主库同时开启流复制 + `archive_command` 是生产标配
- Standby 可以开启 `restore_command`，用归档 WAL 补充流复制的 gap（网络断开后）
- `pg_basebackup --wal-method=stream` 可在备份时同步流式获取 WAL，避免归档滞后

## 5. 合成备份（Synthetic Full）深度

### 5.1 合成全备的读写放大分析

```
假设：
  全量大小 = 1 TB
  每日增量 = 50 GB（5% 变化率）
  连续 7 天增量后做合成全备

合成全备 I/O 成本：
  读：1 TB（全量基线）+ 7 × 50 GB（增量）= 1.35 TB 读
  写：1 TB（新合成全量写出）
  净放大比：(1.35 TB 读 + 1 TB 写) / 50 GB 增量 ≈ 47x

对比传统增量备份（无合成）：
  每日 I/O：50 GB 读（源端）+ 50 GB 写（仓库）= 2x
```

> ⚠️ 合成全备在仓库端产生巨大 I/O。如仓库使用高重删 HDD（随机读性能差），合成时间可能达到 8–12h，与前台备份任务产生严重资源争抢。

### 5.2 Forever-Incremental + 后台合成的调度策略

- **永久增量**：Day1 全量，Day2+ 仅增量 → 备份窗口最短
- **后台合成**：离峰期仓库内合成，不影响日常备份窗口
- 合成触发条件通常是：恢复链深度超过阈值（如 7 天）或保留策略到期
- Veeam 的 Per-VM Backup Chain 隔离了不同 VM 的合成影响范围
- 合成期间恢复请求不应受阻，需要仓库 I/O 隔离或优先级队列

## 6. 不可变副本与勒索软件防护

### 6.1 不可变存储的实现层次

| 实现方式 | 原理 | 可绕过场景 | 典型产品 |
|---------|------|-----------|---------|
| Linux Hardened Repo | 单用途 Linux + `chattr +i` + 禁 SSH root | 拥有物理访问权或 root 的攻击者 | Veeam Hardened Repository |
| S3 Object Lock (WORM) | S3 API 层设置 Retention Period，对象在期限内不可删改 | AWS 账号根凭证泄露 | AWS S3 / MinIO / Wasabi |
| WORM Appliance | 硬件级写保护，固件执行不可改策略 | 物理摧毁设备 | Dell EMC ECS, NetApp SnapLock |
| Air-Gap / Tape Vault | 物理离线，完全断网 | 无法远程攻击 | 磁带库 / 离线磁盘库 |
| 隔离恢复保管库 (CRV) | 网络隔离 + 定期单向数据复制 + 独立身份认证 | 内部人员攻击 | Dell Cyber Recovery Vault |

### 6.2 勒索软件攻击链与防护节点

```
[初始入侵] → [横向移动] → [提权] → [发现备份服务器] → [删除/加密备份] → [加密生产数据] → [勒索]

关键防护节点：

  1. 备份账号与生产网络隔离                    → 防止凭证横向复用
  2. 备份 Console 禁止来自生产网段直接 RDP/SSH  → 防止发现后直接攻击
  3. 不可变仓库 / WORM / Object Lock           → 攻击者无法删除备份
  4. MFA + 4-eyes approval（删除备份需双人审批）→ 防止管理员账号被接管后单人操作
  5. 恢复前的恶意软件扫描                       → 防止带毒恢复
  6. 隔离网络中的 Clean Room 恢复               → 带毒环境下的安全验证
```

### 6.3 Object Lock 模式详解

| 模式 | 含义 | 管理员能否提前删除 | 适用场景 |
|------|------|-----------------|---------|
| Compliance Mode | 在保留期内，任何用户（含根账号）均不可删改 | ✗ 不能 | 高合规要求（金融/医疗） |
| Governance Mode | 有特权权限（`s3:BypassGovernanceRetention`）的用户可删 | ✓ 特权账号可以 | 企业内部策略灵活调整 |
| Legal Hold | 独立于 Retention Period，手动加/解除锁 | 需主动解除 | 诉讼保全 |

## 7. 边界计算（面试设计题必考）

### 7.1 备份窗口计算

```
有效备份吞吐（端到端）= min(
  源端读取速度（受 CBT 提取 + VMDK I/O 限制）,
  Proxy 处理速度（压缩/加密/hash CPU 瓶颈）,
  网络带宽（Proxy → Repository）,
  仓库写入速度（顺序写 IOPS × 块大小）
)

备份窗口 = 总变化量 / 有效吞吐

示例：
  200 台 VM，日均变化 10%，总量 400 TB
  日增量 = 40 TB
  有效吞吐：10 个 Proxy × 1 Gbps × 压缩比 2x ÷ 8 bits = 2.5 GB/s
  备份窗口 ≈ 40 TB / 2.5 GB/s ≈ 4.4 小时

如目标窗口 2 小时 → 需将有效吞吐提升到 5.5 GB/s
  → 增加 Proxy 数量，或升级仓库网络，或减少变化量
```

### 7.2 真实 RTO 计算

```
真实 RTO = 数据传输时间
         + 解密/解压时间
         + 元数据重建时间
         + VM 注册/开机时间
         + 数据库 WAL replay 时间
         + 应用 warm-up
         + DNS/LB 切换
         + 功能验证时间

常见低估：
  - 只算数据还原速度（忽略其余项）
  - 未考虑恢复目标环境资源是否就绪
  - 数据库 WAL replay 对于大库可能需要 1–2 小时
  - 高重删仓库恢复时随机读 IOPS 极差
```

### 7.3 RPO 与备份频率、WAL 归档延迟的关系

```
RPO 上界 = 备份间隔 + 数据传输延迟 + 检测故障时间

对于 PostgreSQL PITR：
  RPO ≈ WAL 归档延迟（通常 1–5 min，取决于 archive_timeout 设置）
  archive_timeout = 60 → 最多丢 60s WAL

对于 VMware 虚拟机：
  RPO = 备份间隔（每小时备份 → RPO 最大 1h）
  CDP 方案 → RPO 可到秒级，但成本大幅提升

降低 RPO 的成本曲线是非线性的：
  1h RPO   → 普通备份策略，成本基准
  5min RPO → 高频备份或日志保护，成本 3–5x
  秒级 RPO → CDP/同步复制，成本 10x+
```

### 7.4 重删率边界

| 数据类型 | 典型重删率 | 原因 |
|---------|-----------|------|
| 模板 VM（相同 OS/应用） | 70–95% | 大量相同基础 OS 块 |
| 通用文件服务器（Office 文档） | 40–60% | 文档结构相似，局部变化 |
| 数据库数据文件（热页面） | 5–20% | 页面频繁随机更新，重复率低 |
| 已压缩/加密文件（zip、AES） | < 5% | 块内容随机化，hash 几乎全不同 |
| 媒体文件（视频/音频） | < 5% | 已压缩，内容唯一 |
| 归档日志（重复操作多） | 30–50% | 相同操作模式重复出现 |

> ⚠️ 重删率是 workload 的统计学结果，不是产品承诺。Vendor 给出的 "20:1 dedup ratio" 都是在特定模板 VM workload 下测量的，不可作为通用估算基准。

## 8. 容灾设计

### 8.1 四种容灾模式量化对比

| 模式 | RPO | RTO | 资源成本 | 切换方式 | 适用场景 |
|------|-----|-----|---------|---------|---------|
| 冷备 Cold | 小时~天级 | 小时~天级 | 低（仅存储） | 手动 | 低优先级、合规归档 |
| 温备 Warm | 分钟~小时 | 15min~2h | 中（预配基础设施） | 半自动 | 中等核心系统 |
| 热备 Hot | 秒~分钟 | 1–15 min | 高（全量镜像资源） | 自动/半自动 | 核心业务系统 |
| 多活 Active-Active | < 1s | < 1 min | 极高（2x+ 资源） | 透明无切换 | 金融交易、核心电商 |

### 8.2 多活架构的数据一致性挑战

- **写冲突**：两个站点同时写同一条记录 → 需要 Conflict Detection + Resolution 策略（Last-Write-Wins / CRDT / 业务层仲裁）
- **读己之写一致性**：用户写 Site A，立即读 Site B → 复制延迟导致读不到自己刚写的数据
- **跨站点事务**：需分布式事务（2PC/Saga），代价是延迟和可用性下降
- **CAP 约束**：多活提升 A（可用性），但在分区（P）场景下必然牺牲 C（强一致性）
- **实践**：大多数"多活"系统实际是**按业务分片**（用户/区域）分流，而非真正全局共享写

### 8.3 演练（DR Drill）的工程要求

- 演练频率：核心系统至少每季度一次，完整切换演练（非只验证数据完整性）
- 演练范围：必须包含真实账号、真实密钥、真实网络路由、真实 DNS 切换——仿真环境演练成功不等于生产可用
- 验收标准：不只是"VM 能启动"，而是"业务请求能正常响应"（端到端功能测试）
- Runbook：每次演练后更新，记录依赖变化（证书、IP、配置变更）
- RTO 测量：从"故障确认"到"业务验收通过"的完整计时，不是从"恢复命令发出"计时

## 9. 面试实战排障

### 9.1 备份窗口越来越长

| 排查点 | 检查方式 | 常见根因 |
|--------|---------|---------|
| 变化量暴涨 | 对比 CBT 返回的 extent 总量 | 应用写放大、数据库全表扫描、swap 频繁 |
| CBT 失效 | 查看是否有 full scan 日志 | VM 重配置、snapshot 链异常 |
| Proxy CPU 打满 | 监控 Proxy CPU 使用率 | 并发 job 过多、压缩比设置过高 |
| 仓库 I/O 争抢 | 检查 Repo I/O 队列深度 | 合成全备与备份任务并行执行 |
| 网络拥塞 | 监控 Proxy → Repo 带宽利用率 | 共享备份网络与业务网络 |
| Snapshot consolidate 耗时 | 查看 consolidate 日志时长 | Delta 链过长、存储 I/O 差 |

### 9.2 恢复特别慢

- 恢复点链过长 → 需要读取大量增量块 → 考虑合成全备频率
- 仓库为高重删 HDD → 恢复时需大量随机读（拼装 chunk）→ 用 SSD 缓存层加速
- 对象存储回源 → GET 延迟高 + API 限速 → 检查 Tier Policy 是否误将热数据归档
- 数据库 WAL replay 长 → 与 base backup 距离过远 → 增加备份频率
- 解密瓶颈 → 检查 Proxy CPU 和 AES-NI 加速是否启用
- 目标环境资源不足 → 内存/CPU 限制 → 确认恢复目标与生产规格一致

### 9.3 设计题：200 台 VM 企业级备份方案

```
需求澄清（必须先问）：
  - RPO / RTO 要求？（核心系统 vs 边缘系统差异化）
  - 是否有勒索软件防护需求？
  - 存储层是否支持 SAN/HotAdd？
  - 是否有合规/保留期要求？

分层设计：
  ┌────────────────────────────────────────────┐
  │ 保护层         │ 技术选型         │ RPO    │
  ├────────────────────────────────────────────┤
  │ 快速回滚       │ VMware Snapshot  │ 0      │
  │ 日常备份       │ CBT 增量 + 合成全 │ 1–4h   │
  │ 不可变副本     │ Hardened Repo    │ 1–4h   │
  │ 异地归档       │ S3 Object Lock   │ 1 day  │
  │ 离线保管       │ Air-gap / Tape   │ 1 week │
  └────────────────────────────────────────────┘

监控指标（不只看 job 成功率）：
  - 恢复点有效率（能成功 restore 的比例）
  - 不可变副本覆盖率
  - WAL 归档延迟（对 DB）
  - 演练 RTO 达标率
```

### 9.4 勒索软件场景恢复步骤

1. 隔离生产网络，防止二次扩散
2. 确认备份服务器是否也被感染（检查 catalog 完整性、不可变副本是否被篡改）
3. 识别最早未感染的恢复点（勒索软件通常潜伏 2–6 周后才触发加密）
4. 在隔离 Clean Room 环境先恢复副本，运行恶意软件扫描
5. 确认干净后，再向生产环境迁移
6. 修复漏洞、轮换所有凭证，重建备份链

## 10. 厂商技术差异

| 厂商 | 重删架构 | VMware 集成深度 | 不可变能力 | 核心技术标签 |
|------|---------|--------------|-----------|------------|
| Veeam | 目标端（Repo 内）+ 源端 WAN 加速 | 最深（VADP 原生） | Linux Hardened Repo + S3 Lock | 永久增量、SureBackup、HotAdd 优化 |
| Commvault | IntelliSnap + 全局重删 | 深，兼顾异构 | MediaAgent 级 WORM + 云对象锁 | 策略编排复杂度、异构 workload 广度 |
| Rubrik | 集群内全局重删（自研文件系统） | 深，含快照编排 | Immutable 架构（所有快照不可变） | SLA 策略驱动、威胁分析、Cloud Vault |
| Cohesity | WebScale 全局重删 | 深 | DataLock（WORM） | 二级存储融合、AI 威胁检测 |
| Data Domain (Dell) | 自研 DDFS + DD Boost | 需 NDMP/DD Boost 插件 | Retention Lock + CyberSense | 目标端重删最成熟、大型企业场景 |
| NetBackup (Veritas) | MSDP（Media Server Dedup Pool） | 深，大规模支持好 | WORM Storage + Immutable Image | 异构大规模、长期企业演进 |

> 💡 面试中不要答"谁最好"。应按：虚拟化集成深度 / 重删位置（源端 vs 目标端）/ 不可变能力实现层次 / 工作负载覆盖广度来横向比较。

## 核心原则总结

> ⚡ 备份的价值不在于「备了多少」，而在于「出事后能在目标时间内，从未被污染的副本以可信一致性恢复业务」。

- **VMware 备份核心**：CBT 识别增量 + 快照管理 + Proxy 传输模式选择，瓶颈在 consolidate 和仓库 I/O
- **重删核心**：CDC 变长分块 + 双 hash 碰撞防护 + 内存索引是性能瓶颈 + GC 是资源风险点
- **PostgreSQL 核心**：Base Backup + WAL 持续归档 + PITR，WAL 链完整性是唯一核心约束
- **合成全备核心**：省源端窗口，代价是仓库端 I/O 放大，高重删 + 合成 = 仓库资源双重压力
- **不可变核心**：分层（Linux/S3/WORM/Air-gap），Object Lock Compliance 模式最强但灵活性最差
- **RTO 核心**：真实 RTO 包含数据传输 + 重建 + 应用启动 + 验证，演练是唯一可信的 RTO 证明
