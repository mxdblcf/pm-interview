> 适用场景：云存储 / 基础设施 / IaaS 方向 PM 面试  
> 内容风格：原理理解 + 业务视角 + 图示说明 + 发展趋势

## 一、什么是块存储？先从最直白的比喻说起

### 1.1 三种存储形态的区别

存储产品有三种核心形态，面试中经常被要求区分：

```
┌─────────────────────────────────────────────────────────────┐
│                    三种存储形态对比                           │
├──────────────┬──────────────────┬──────────────────────────┤
│   块存储      │    文件存储       │       对象存储            │
│  Block       │    File          │       Object             │
├──────────────┼──────────────────┼──────────────────────────┤
│ 像一块未格式  │ 像一个网络共享    │ 像一个超大的网盘           │
│ 化的"裸硬盘"  │ 文件夹，直接挂载  │ 通过 URL/API 访问         │
├──────────────┼──────────────────┼──────────────────────────┤
│ 自己决定格式  │ 已有目录和文件    │ Key-Value 平铺存放        │
│ 化方式        │ 层级结构         │ 无层级限制                │
├──────────────┼──────────────────┼──────────────────────────┤
│ 最高性能      │ 多机共享方便      │ 海量数据、低成本           │
├──────────────┼──────────────────┼──────────────────────────┤
│ 典型产品：    │ 典型产品：        │ 典型产品：                │
│ AWS EBS      │ AWS EFS / NAS    │ AWS S3 / 阿里 OSS         │
│ 阿里云云盘    │ 阿里云 NAS       │ 腾讯云 COS                │
└──────────────┴──────────────────┴──────────────────────────┘
```

**一句话记住块存储的本质**：  
> 块存储 = 给云服务器用的"虚拟硬盘"，性能最高，数据库首选。

---

### 1.2 块存储的工作原理（图示）

块存储把磁盘切成固定大小的"块"（通常 4KB），每块有唯一编号，像一排有编号的储物格：

```
逻辑块设备（用户看到的）
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ 0  │ 1  │ 2  │ 3  │ 4  │ 5  │ 6  │ 7  │  ← 块编号（LBA）
└────┴────┴────┴────┴────┴────┴────┴────┘
         ↓ 映射关系（集中式：直接映射；分布式：CRUSH/哈希）
物理存储介质
┌────┬────┬────┬────┬────┬────┬────┬────┐
│实际│数据│存放│在这│里面│不一│定连│续  │
└────┴────┴────┴────┴────┴────┴────┴────┘
```

**I/O 请求完整路径（从应用到磁盘）：**

```
用户应用（数据库 / 文件系统）
        ↓ 读写请求
   操作系统缓存层
        ↓ 缓存未命中时
   块设备驱动层（把请求打包）
        ↓ 本地 or 网络传输
   存储介质（HDD / SSD / NVMe）
        ↓ 返回数据
   应用拿到结果
```

> **PM 视角理解**：每一层都会带来延迟，层数越少、越靠近硬件，性能越高。  
> 本地 NVMe 最快（微秒级），云网络块存储次之（毫秒级），对象存储最慢（数十毫秒）。

## 二、块存储发展历史

### 2.1 发展时间线

```
1950s-1970s          1980s-1990s         2000s              2010s              2020s至今
    │                    │                  │                  │                  │
    ▼                    ▼                  ▼                  ▼                  ▼
磁带 / 早期磁盘      SCSI 协议 / RAID     SAN / NAS 普及     云存储崛起          全闪存 / NVMe
本地直连              集中式存储阵列        网络化存储          分布式块存储         超低延迟时代
大型机独占            企业存储主流          iSCSI / FC         EBS / Ceph         NVMeoF / SPDK
```

### 2.2 各阶段详细说明

**第一阶段（1950s-1990s）：本地直连时代**

- 磁盘直接连接在服务器上（DAS，Direct Attached Storage）
- 一台服务器一块磁盘，互相不共享
- 问题：磁盘坏了，数据全丢；扩容难，性能受单机限制

**第二阶段（1990s-2000s）：集中式存储阵列时代**

- 出现了专用的"存储阵列"设备（EMC、NetApp、IBM 等厂商）
- RAID 技术发明：多块磁盘组合，既提升性能又能容错
- SAN（存储区域网络）出现：服务器通过专用网络访问存储阵列
- 问题：设备昂贵、扩展性差、存在单点瓶颈

**第三阶段（2006年-2015年）：云存储 / 分布式时代**

- AWS 2006 年推出 EBS，块存储首次进入云端
- Ceph 开源项目发布，分布式块存储走向主流
- 存储资源池化，按需付费，弹性扩展
- 问题：网络延迟比本地存储高，早期稳定性有挑战

**第四阶段（2015年-至今）：全闪存 / 超融合时代**

- NVMe SSD 普及，存储延迟从毫秒级降到微秒级
- 超融合架构（计算和存储跑在同一台服务器）流行
- 多云、跨云存储需求出现
- AI 训练带来海量存储新需求

## 三、集中式块存储

### 3.1 什么是集中式块存储

集中式块存储 = 把所有存储资源集中在一台（或一组）专用存储设备上，服务器通过网络访问。

```
              ┌─────────────────────────────┐
              │        存储阵列              │
              │  ┌──────┐  ┌──────┐        │
              │  │磁盘组1│  │磁盘组2│  ...  │
              │  └──────┘  └──────┘        │
              │      统一管理、统一调度       │
              └──────────────┬──────────────┘
                             │ 专用存储网络（FC/iSCSI）
         ┌───────────────────┼───────────────────┐
         ↓                   ↓                   ↓
    服务器 A            服务器 B            服务器 C
```

### 3.2 核心技术：SAN（存储区域网络）

SAN 是连接服务器和存储阵列的专用高速网络，不走普通以太网：

```
┌──────────┐        ┌──────────────┐        ┌──────────────┐
│ 服务器 A  │        │              │        │              │
│          │──FC────│   SAN 交换机  │────────│  存储阵列    │
│ 服务器 B  │        │  （高速专网）  │        │ （EMC/NetApp）│
│          │──iSCSI─│              │        │              │
└──────────┘        └──────────────┘        └──────────────┘
```

**两种 SAN 协议对比：**

```
┌───────────────┬──────────────────────┬──────────────────────┐
│               │    FC（光纤通道）      │   iSCSI（IP 网络）   │
├───────────────┼──────────────────────┼──────────────────────┤
│ 传输介质       │ 专用光纤               │ 普通以太网            │
│ 速度           │ 16/32 Gbps           │ 10/25 Gbps           │
│ 延迟           │ 极低（微秒）           │ 较低（微秒-毫秒）     │
│ 成本           │ 高（需专用设备）        │ 低（复用现有网络）    │
│ 适用场景       │ 金融、电信、超大规模    │ 中小企业、通用场景    │
└───────────────┴──────────────────────┴──────────────────────┘
```

### 3.3 核心技术：RAID（磁盘冗余阵列）

RAID = 把多块磁盘组合成一个逻辑盘，提升性能或可靠性（或两者兼顾）：

```
RAID 0（条带化，纯性能）：
  数据 A → 磁盘1  │  数据 B → 磁盘2  （并行写，速度翻倍，无容错）
  
RAID 1（镜像，纯容错）：
  数据 A → 磁盘1  
  数据 A → 磁盘2  （完整备份，坏一块没事，容量利用率 50%）
  
RAID 5（条带+奇偶校验）：
  磁盘1: 数据A │ 磁盘2: 数据B │ 磁盘3: 校验P（A+B）
  坏一块可恢复，容量利用率 (N-1)/N，主流选择
  
RAID 10（先镜像再条带，性能+容错）：
  [磁盘1+磁盘2 互为镜像] + [磁盘3+磁盘4 互为镜像]
  数据库生产环境推荐
```

**PM 要记住的结论**：
- RAID 0：赌性能，数据无保护，测试环境用
- RAID 1：最简单的容错，成本高
- RAID 5：中小企业最常用，**大容量磁盘重建期间有二次故障风险**
- RAID 10：数据库标配，性能和安全兼顾

### 3.4 集中式块存储的优缺点

```
优点：
✅ 性能稳定，延迟极低（本地访问或专网）
✅ 管理集中，运维成本低
✅ 成熟技术，稳定性高（金融行业 30 年验证）
✅ 快照、复制等高级功能齐全

缺点：
❌ 扩展性差（纵向扩容，存储阵列有上限）
❌ 单点风险（存储阵列是核心，坏了全挂）
❌ 成本高（专用硬件，EMC 动辄数百万）
❌ 跨地域复制复杂
```


## 四、分布式块存储

### 4.1 什么是分布式块存储

分布式块存储 = 把普通服务器的硬盘组成一个"存储资源池"，通过软件实现高性能、高可靠的块存储服务。

```
                    ┌──────────────────────────────────────┐
                    │        分布式存储集群                  │
   ┌─────────┐      │  ┌─────────┐  ┌─────────┐           │
   │ 应用服务器│      │  │存储节点1│  │存储节点2│   ...     │
   │         │─────→│  │ /磁盘A  │  │ /磁盘B  │           │
   │ 应用服务器│      │  └─────────┘  └─────────┘           │
   └─────────┘      │       ↕ 数据自动均衡、自动复制           │
                    │  ┌─────────┐  ┌─────────┐           │
                    │  │存储节点N│  │存储节点M│           │
                    │  └─────────┘  └─────────┘           │
                    └──────────────────────────────────────┘
```

**核心思想**：不再依赖昂贵的专用硬件，用软件解决可靠性和性能问题。

### 4.2 分布式块存储的核心机制

**① 数据分片（Striping）**

一份数据被切成多个小块，分散存放在不同节点上，读写时并行操作：

```
原始数据（100MB）
       ↓ 切分
 ┌──────────────────────────────────────┐
 │ 块A(25MB) │ 块B(25MB) │ 块C(25MB) │ 块D(25MB) │
 └─────┬─────┴─────┬─────┴─────┬─────┴─────┬─────┘
       ↓           ↓           ↓           ↓
   节点1         节点2         节点3        节点4
                  （四路并行 I/O，性能是单节点 4 倍）
```

**② 多副本（Replication）**

每份数据同时存多份，任何一个节点坏掉都不丢数据：

```
写入数据 X
    ↓ 自动复制（3副本）
节点1: X的副本1  ← 主副本
节点2: X的副本2  ← 备副本1（不同机架）
节点3: X的副本3  ← 备副本2（不同机房/可用区）

节点1 故障 → 自动从节点2/3 读取，用户无感知
```

**③ 纠删码（Erasure Code，更省空间的容错方案）**

纠删码是比多副本更节省空间的方案，用数学计算代替"完整备份"：

```
原始数据：4 块（D1 D2 D3 D4）
纠删码计算出：2 块校验数据（P1 P2）
共存 6 块，分散在 6 个节点

节点宕机 2 个 → 剩余 4 块通过数学运算恢复出全部数据
    
空间利用率对比：
  3副本：存 100GB 需要 300GB 空间（利用率 33%）
  4+2纠删码：存 100GB 需要 150GB 空间（利用率 67%）
```

> **PM 注意**：纠删码省空间，但重建时计算量更大，适合冷数据；热数据仍推荐多副本。

**④ 一致性哈希 / CRUSH 算法（数据如何找到存放位置）**

```
写入数据时，系统需要决定：这块数据放在哪个节点？

传统中心化方式（有单点瓶颈）：
  数据 → 查询元数据服务器 → 得到节点地址 → 写入

分布式哈希方式（Ceph CRUSH 算法）：
  数据 → 本地计算哈希 → 直接得出目标节点
  （无需查询中心节点，彻底消除元数据瓶颈）
```

### 4.3 云端分布式块存储：以 AWS EBS 为例

EBS（Elastic Block Store）是最典型的云端分布式块存储产品：

```
EC2 虚拟机
    ↓ 虚拟网卡（NVMe over Network）
┌─────────────────────────────────────────┐
│              EBS 服务                    │
│  控制平面：管理卷的元数据、挂载关系、权限   │
│  数据平面：分布式存储节点集群              │
└─────────────────────────────────────────┘
    ↓ 数据自动 3 副本跨可用区存储
物理存储节点（用户不可见）
```

**EBS 核心产品形态：**

```
┌──────────────┬───────────────┬────────────────────────────┐
│   产品类型    │   底层介质     │   典型用途                  │
├──────────────┼───────────────┼────────────────────────────┤
│ gp3（通用）   │ NVMe SSD      │ 大多数工作负载，性价比最高    │
│ io2（高性能） │ NVMe SSD      │ 数据库、低延迟要求            │
│ st1（吞吐型） │ HDD           │ 大数据、日志顺序读写          │
│ sc1（冷存储） │ HDD           │ 归档备份，访问频率低          │
└──────────────┴───────────────┴────────────────────────────┘
```

**快照（Snapshot）机制——PM 必懂：**

```
初始状态：
  卷数据 [A B C D E F]
         ↓ 创建快照1（全量，存入 S3）
        Snap1

修改数据 B、E 后：
  卷数据 [A B' C D E' F]
         ↓ 创建快照2（仅存变化的 B、E，增量）
        Snap2（只含 B→B'，E→E' 的变化）

恢复到 Snap1：还原 B' 为 B，E' 为 E
```

> 增量快照的好处：存储成本低（只存变化），备份速度快。

### 4.4 开源分布式块存储：Ceph RBD

Ceph 是最主流的开源分布式存储系统，RBD（RADOS Block Device）是其块存储模块：

```
用户看到的：一块虚拟块设备（/dev/rbd0）
              ↓ 被切成 4MB 的对象
              ↓ CRUSH 算法计算每个对象应放在哪
              ↓
         ┌────────────────────────────────┐
         │          OSD 集群              │
         │  OSD1   OSD2   OSD3  ...      │
         │ (每个OSD管理一块物理磁盘)        │
         └────────────────────────────────┘
```

**Ceph 适用场景：**
- 私有云、混合云（OpenStack 默认存储后端）
- 对成本敏感、需要开源定制的大型互联网公司
- 国内主流：阿里云、腾讯云早期存储系统均有 Ceph 影子

### 4.5 分布式块存储的优缺点

```
优点：
✅ 弹性扩展（加节点即可扩容，无上限）
✅ 无单点故障（任意节点宕机不影响服务）
✅ 成本低（用普通 x86 服务器，不需要专用硬件）
✅ 云原生友好，按需付费

缺点：
❌ 网络延迟引入（比本地存储多 1-5ms）
❌ 系统复杂，运维门槛高
❌ 一致性保证有挑战（多副本同步问题）
❌ 性能受网络带宽限制
```

## 五、集中式 vs 分布式：横向对比

```
┌──────────────────┬─────────────────────┬─────────────────────┐
│   维度            │    集中式块存储       │    分布式块存储       │
├──────────────────┼─────────────────────┼─────────────────────┤
│ 延迟              │ 极低（μs 级）         │ 较低（ms 级）         │
│ 扩展性            │ 有上限（纵向扩展）     │ 近乎无限（横向扩展）   │
│ 可靠性            │ 高（专业设备）         │ 高（软件多副本）       │
│ 成本              │ 高（专用硬件）         │ 低（通用服务器）       │
│ 运维复杂度        │ 低（厂商托管）         │ 高（需要存储团队）     │
│ 单点风险          │ 存在（阵列是核心）     │ 无（节点可任意失效）   │
│ 适合规模          │ 中小规模，稳定负载     │ 大规模，弹性负载       │
│ 典型客户          │ 银行、电信、政府       │ 互联网、云厂商         │
│ 代表产品          │ EMC、NetApp、华为     │ AWS EBS、Ceph、Rook   │
└──────────────────┴─────────────────────┴─────────────────────┘
```

**选型决策树：**

```
需要块存储？
    ↓
延迟要求 < 100μs？
    ├─ 是 → 本地 NVMe 或高端 SAN 阵列
    └─ 否 → 继续判断
         ↓
规模是否大 / 需要弹性扩展？
    ├─ 是 → 分布式块存储（云盘 / Ceph）
    └─ 否 → 看预算
              ├─ 预算充足 → 集中式存储阵列
              └─ 预算有限 → 分布式（Ceph 自建）
```

## 六、关键技术概念（面试高频词汇）

### 6.1 IOPS、吞吐量、延迟——三大核心指标

```
┌─────────┬────────────────────────────────────────────────┐
│ 指标     │  含义 & PM 理解                                  │
├─────────┼────────────────────────────────────────────────┤
│ IOPS    │ 每秒完成的 I/O 操作次数                            │
│         │ 类比：收银台每秒处理多少位顾客                       │
│         │ 数据库（随机小文件）最看重 IOPS                     │
├─────────┼────────────────────────────────────────────────┤
│ 吞吐量   │ 每秒传输的数据量（MB/s）                           │
│         │ 类比：水管每秒流过多少水                            │
│         │ 大数据、视频处理（顺序大文件）最看重吞吐量             │
├─────────┼────────────────────────────────────────────────┤
│ 延迟     │ 一次 I/O 从发出请求到收到响应的时间                  │
│         │ 类比：收银台结账等待时长                             │
│         │ 在线交易系统最看重延迟                              │
└─────────┴────────────────────────────────────────────────┘

各类存储典型指标：
┌─────────────┬──────────────┬───────────┬────────────┐
│ 存储类型     │  IOPS        │ 吞吐量     │ 延迟       │
├─────────────┼──────────────┼───────────┼────────────┤
│ 机械硬盘     │ 100-200      │ 150MB/s   │ 5-10ms     │
│ SATA SSD    │ 8-10万       │ 550MB/s   │ 50-100μs   │
│ NVMe SSD    │ 50-100万     │ 7000MB/s  │ 10-50μs    │
│ EBS gp3     │ 最高 1.6万   │ 1000MB/s  │ 个位数 ms  │
│ EBS io2     │ 最高 25.6万  │ 4000MB/s  │ 亚毫秒     │
└─────────────┴──────────────┴───────────┴────────────┘
```

### 6.2 快照 vs 备份 vs 复制

```
┌──────────┬────────────────────────────────────────────────┐
│ 快照      │ 某一时刻的数据状态记录（秒级创建，增量存储）         │
│ Snapshot │ 用途：回滚误操作、创建测试环境                      │
│          │ 类比：游戏存档，随时回到某个节点                     │
├──────────┼────────────────────────────────────────────────┤
│ 备份      │ 完整数据的拷贝（通常到异地/不同介质）                │
│ Backup   │ 用途：灾难恢复，应对勒索病毒                        │
│          │ 类比：把游戏存档复制到另一台电脑                     │
├──────────┼────────────────────────────────────────────────┤
│ 复制      │ 实时/近实时把数据同步到另一套存储                   │
│ Replica  │ 用途：高可用、异地容灾                             │
│          │ 类比：游戏联机同步，两台机器数据实时一致              │
└──────────┴────────────────────────────────────────────────┘
```

### 6.3 高可用与容灾（RTO / RPO）

```
┌─────┬─────────────────────────────────────────────────────┐
│ RPO │ 恢复点目标（Recovery Point Objective）                │
│     │ = 最多能接受丢失多长时间的数据                          │
│     │ RPO = 0：数据不允许任何丢失（实时同步）                 │
│     │ RPO = 1h：最多丢失 1 小时数据（每小时备份一次即可）      │
├─────┼─────────────────────────────────────────────────────┤
│ RTO │ 恢复时间目标（Recovery Time Objective）               │
│     │ = 发生故障后多长时间内必须恢复业务                       │
│     │ RTO = 5min：5 分钟内必须恢复（需要热备）                │
│     │ RTO = 24h：24 小时内恢复（冷备即可）                   │
└─────┴─────────────────────────────────────────────────────┘

理解图示：
  业务连续时间轴 ─────────────────────────────────→
               ↑ 上次备份          ↑ 故障发生    ↑ 业务恢复
               │←────── RPO ───────→│            │
                                    │←── RTO ────→│
```

### 6.4 存储的"脑裂"问题

```
问题场景：
┌─────────┐   网络中断   ┌─────────┐
│ 存储节点A│─────✗───────│ 存储节点B│
└─────────┘             └─────────┘
    ↑                        ↑
  客户端1                  客户端2
  （写入数据X）             （写入数据Y）

网络恢复后：节点A有X，节点B有Y，数据不一致 → 脑裂！

解决方案：
  多数派仲裁（Quorum）：写操作需要超过半数节点确认才算成功
  Raft/Paxos 共识算法：分布式系统的主从选举协议
  Fencing 机制：确认旧主节点已经"死透"再切换新主
```

## 七、未来发展趋势

### 7.1 近期趋势（2024-2027）

**① NVMe 全面替代 SATA/SAS**

```
历史路径：HDD → SATA SSD → NVMe SSD
当前状态：企业存储正在全面迁移到 NVMe
影响：存储延迟从毫秒级降至微秒级，IOPS 提升 10-100 倍
```

**② NVMe over Fabrics（NVMeoF）：让远端存储接近本地延迟**

```
传统网络存储：主机 ──以太网──→ 存储阵列  （延迟 1-5ms）
NVMeoF：     主机 ──RDMA/TCP→ 远端NVMe  （延迟 <100μs）

意义：让分布式存储的延迟逼近本地 SSD，消弭本地 vs 网络存储的界限
```

**③ 存储与计算融合（超融合架构）**

```
传统架构：计算服务器 + 独立存储阵列（两套硬件，专线连接）

超融合架构（HCI）：
┌────────────────────────────────────────┐
│   超融合节点1  │  超融合节点2  │  节点N │
│  CPU + 内存   │  CPU + 内存  │  ...  │
│  + 本地 NVMe  │  + 本地 NVMe │       │
│   运行虚拟机   │   运行虚拟机  │       │
└────────────────────────────────────────┘
     ↕ 软件定义存储，本地磁盘共享给集群所有节点

代表产品：VMware vSAN、Nutanix、华为 FusionCube
优势：部署简单、弹性扩展、成本低
```

**④ 云原生存储（Kubernetes 时代的块存储）**

```
容器时代的存储挑战：
  容器本身是"无状态"的，但数据库等应用需要持久化存储
  容器随时可能在任意节点启动 → 存储必须跟着走

解决方案：CSI（Container Storage Interface）标准
  ┌─────────────────────────────────────────┐
  │  Kubernetes Pod（容器）                  │
  │   ↕ CSI 接口（统一标准）                 │
  │  块存储插件（AWS EBS / Ceph / Portworx） │
  └─────────────────────────────────────────┘
  Pod 迁移到新节点 → 块存储自动重新挂载
```

### 7.2 中期趋势（2027-2030）

**⑤ AI 训练专用存储**

```
AI 训练的存储特点：
  - 数据量极大（TB-PB 级训练集）
  - 顺序读取为主（批量加载数据）
  - GPU 集群并发读，对带宽要求极高
  - 检查点（Checkpoint）保存需要高 IOPS 写入

新兴产品方向：
  GPU Direct Storage（跳过 CPU，GPU 直接读存储）
  高带宽并行文件系统 + 块存储混合架构
  国内：阿里灵骏、华为昇腾存储生态
```

**⑥ 计算存储（Computational Storage）**

```
传统方式：数据从存储读出 → CPU 计算 → 结果写回存储（大量数据搬运）

计算存储：在 SSD/存储控制器内嵌入计算单元
  数据在存储设备内部完成过滤/压缩/加密
  只返回"有用"的结果给 CPU
  减少 80% 以上的数据传输量

典型场景：数据库下推过滤（近数据计算）
```

**⑦ 存储即服务（STaaS）深化**

```
从"买存储设备"到"按 GB/IOPS 订阅存储服务"
  - 按实际消耗付费，无需容量规划
  - 存储厂商负责硬件升级，用户感知不到
  - 多云统一存储层（Netapp、Pure Storage 的新模式）
```

### 7.3 长期趋势（2030+）

**⑧ 持久化内存（Persistent Memory / PMem）**

```
内存（DRAM）：极快（纳秒），掉电丢数据
存储（SSD） ：较快（微秒），掉电不丢

持久化内存（如 Intel Optane PMem）：
  速度接近 DRAM，掉电不丢数据
  彻底模糊"内存"与"存储"的边界
  
当前状态：Intel 已停产 Optane PMem，但技术方向被证明正确
         三星、SK Hynix 仍在研发下一代产品
```

**⑨ 光子存储、DNA 存储（探索阶段）**

- 光子存储：用光信号代替电信号，速度更快、能耗更低
- DNA 存储：每克 DNA 可存 215 PB，适合超长期归档（理论阶段）

## 八、产品经理面试实战问题

### 8.1 常见面试题及回答思路

**Q1：如果让你设计一个云盘产品，你会如何规划核心功能？**

```
思路框架：
  1. 用户需求层面
     - 谁在用？（开发者/DBA/企业 IT）
     - 用来做什么？（数据库/文件系统/日志）
     - 最痛的需求？（性能/成本/可靠性）

  2. 核心功能规划
     - 基础：创建/删除/挂载/卸载/扩容
     - 数据保护：快照、跨区复制、自动备份
     - 性能：分级存储（热/温/冷）、IOPS 限速（QoS）
     - 运维：监控报警、使用量分析

  3. 商业模式
     - 按容量计费 vs 按 IOPS 计费 vs 组合
     - 预留实例（折扣）vs 按需（灵活）
```

**Q2：用户反馈云盘性能差，你作为 PM 怎么处理？**

```
结构化排查思路：
  ① 量化问题：是 IOPS 打满？延迟高？吞吐不足？
  ② 定位层次：
     应用层（SQL 没有索引？）→
     OS 层（文件系统配置）→
     云盘配置（卷类型是否匹配场景）→
     实例层（EC2 的 EBS 带宽是否达到上限）
  ③ 短期解决：升级卷类型 / 增加 IOPS 配额
  ④ 长期优化：应用架构改进、读写分离、缓存层
```

**Q3：块存储 vs 对象存储，什么场景用哪个？**

```
块存储选型：
  - 需要文件系统（运行数据库、操作系统）
  - 要求低延迟随机读写
  - 需要 POSIX 接口（标准文件操作）

对象存储选型：
  - 存海量非结构化数据（图片、视频、日志）
  - 不需要随机修改（只写一次，多次读）
  - 需要全球 CDN 分发
  - 极低成本归档
```

**Q4：解释一下 RPO 和 RTO，并给出一个实际案例**

```
案例：某电商平台存储容灾规划

  业务等级          RPO     RTO     方案
  ─────────────────────────────────────────────
  核心交易数据库    0（零）  5min    实时同步 + 热备切换
  用户行为日志      1小时    2小时   每小时快照备份
  历史订单归档      24小时   24小时  每日备份到冷存储
  图片/视频资源     72小时   48小时  对象存储异地备份

PM 的职责：根据业务价值分级，平衡容灾成本与 SLA 要求
```

**Q5：Ceph 和 EBS 有什么区别，什么时候选自建 Ceph？**

```
EBS（托管服务）：
  ✅ 免运维，开箱即用
  ✅ SLA 保障（99.999%）
  ❌ 成本高，数据在云厂商
  ❌ 定制化能力弱

自建 Ceph：
  ✅ 完全掌控，数据自主
  ✅ 大规模下成本极低（阿里/腾讯的早期选择）
  ❌ 需要专业运维团队
  ❌ 踩坑成本高，稳定性建设周期长

选自建 Ceph 的条件：
  - 数据量 PB 级以上（规模效益显现）
  - 有能力的基础设施团队（5人以上存储专家）
  - 对数据主权有强需求（金融、政务私有云）
  - 有定制化需求（特殊硬件、特殊协议）
```

## 九、一页纸速记表（面试前最后复习）

```
【块存储本质】
  虚拟硬盘 → 固定大小的块 → 最高性能 → 数据库首选

【集中式 vs 分布式】
  集中式：低延迟、稳定、贵、扩展难 → 金融银行
  分布式：弹性扩展、便宜、有网络延迟 → 云计算/互联网

【三大性能指标】
  IOPS = 每秒操作次数（数据库看这个）
  吞吐 = 每秒传输量（大数据看这个）
  延迟 = 单次响应时间（交易系统看这个）

【快照原理】
  COW（写时复制）→ 增量存储 → 秒级创建

【可靠性保障】
  多副本（简单但贵）vs 纠删码（省空间但重建慢）

【容灾指标】
  RPO = 能接受丢多少数据
  RTO = 多久恢复业务

【未来方向】
  NVMe 普及 → NVMeoF 远端接近本地 → 超融合 HCI
  云原生 CSI → AI 训练存储 → 计算存储融合

【主要厂商】
  集中式：EMC、NetApp、华为 OceanStor
  云端分布式：AWS EBS、阿里云云盘、腾讯云 CBS
  开源分布式：Ceph（私有云首选）
  超融合：VMware vSAN、Nutanix
```
