# 对象存储 (Object Storage) —— 底层原理、架构设计与公有云实战

## 一、存储模型对比：块存储 vs 文件存储 vs 对象存储

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                        三种存储模型对比                                        │
│                                                                              │
│  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐            │
│  │   块存储           │  │   文件存储         │  │   对象存储         │            │
│  │   Block Storage   │  │   File Storage   │  │   Object Storage │            │
│  ├──────────────────┤  ├──────────────────┤  ├──────────────────┤            │
│  │                  │  │                  │  │                  │            │
│  │  ┌──┬──┬──┬──┐   │  │  /home/           │  │ ┌──────────────┐ │            │
│  │  │B0│B1│B2│B3│   │  │  ├── user/        │  │ │ Key(ID)      │ │            │
│  │  ├──┼──┼──┼──┤   │  │  │   ├── a.txt    │  │ │ Data(Blob)   │ │            │
│  │  │B4│B5│B6│B7│   │  │  │   └── b.jpg    │  │ │ Metadata     │ │            │
│  │  └──┴──┴──┴──┘   │  │  └── logs/        │  │ └──────────────┘ │            │
│  │                  │  │      └── sys.log   │  │                  │            │
│  │  固定大小块       │  │  树状层级结构       │  │  扁平命名空间     │            │
│  │  无元数据         │  │  POSIX 语义        │  │  丰富自定义元数据 │            │
│  │  低延迟           │  │  支持锁/权限       │  │  HTTP API 访问   │            │
│  └──────────────────┘  └──────────────────┘  └──────────────────┘            │
│                                                                              │
│  适用场景:                                                                    │
│  数据库、虚拟机磁盘    NAS共享、日志目录      图片/视频/备份/数据湖             │
│  低延迟 IOPS 敏感      文件共享/编辑协作      海量非结构化数据                  │
└──────────────────────────────────────────────────────────────────────────────┘
```

| 维度 | 块存储 | 文件存储 | 对象存储 |
|------|--------|---------|---------|
| **数据单元** | 固定大小块 (512B/4KB) | 文件 + 目录树 | Object (Key + Data + Metadata) |
| **访问协议** | SCSI / NVMe / iSCSI | NFS / SMB / CIFS | HTTP REST (S3 API) |
| **寻址方式** | LBA (逻辑块地址) | 路径 `/dir/file` | 唯一 Key (桶 + 对象名) |
| **元数据** | 无（文件系统管理） | 有限（POSIX attrs） | 丰富（自定义 KV） |
| **一致性** | 强一致 | 强一致 | 最终一致（部分操作） |
| **可扩展性** | 受限（需 RAID/LVM） | 中等 | 极强（EB 级） |
| **典型延迟** | 亚毫秒 | 毫秒级 | 10-100ms |
| **典型产品** | EBS / Ceph RBD | EFS / NAS / CephFS | S3 / GCS / Azure Blob / MinIO |

> **面试关键点**：对象存储牺牲了低延迟和 POSIX 语义（不支持 `seek()`、部分修改），换来了**近乎无限的扩展性**和**丰富的元数据管理**能力。

## 二、对象存储核心数据模型

### 2.1 Object 的组成

```
┌─────────────────────────────────────────────────────────┐
│                    Object (对象)                          │
│                                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Key (唯一标识符)                                 │    │
│  │  例: "images/2024/photo-001.jpg"                 │    │
│  │  ★ 扁平命名空间，"/" 只是 Key 的一部分，非真实目录  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Data (数据体 / Blob)                             │    │
│  │  二进制数据，不可部分修改 (Immutable-ish)           │    │
│  │  最大 5TB (S3) / 5TB (GCS) / 4.75TB (Azure)      │    │
│  └─────────────────────────────────────────────────┘    │
│                                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Metadata (元数据)                                │    │
│  │  ┌───────────────────┬─────────────────────────┐ │    │
│  │  │ System Metadata   │ User Metadata           │ │    │
│  │  ├───────────────────┼─────────────────────────┤ │    │
│  │  │ Content-Type      │ x-amz-meta-author       │ │    │
│  │  │ Content-Length    │ x-amz-meta-project      │ │    │
│  │  │ ETag (MD5/CRC)   │ x-amz-meta-version      │ │    │
│  │  │ Last-Modified     │ ... (任意 KV)            │ │    │
│  │  │ Storage-Class     │                         │ │    │
│  │  └───────────────────┴─────────────────────────┘ │    │
│  └─────────────────────────────────────────────────┘    │
│                                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │  ACL / Policy (访问控制)                          │    │
│  │  Bucket Policy / IAM / Pre-signed URL            │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

### 2.2 Bucket (桶) 的概念

```
Account
 ├── Bucket: "my-app-images"     ← 全局唯一命名
 │    ├── Object: "avatar/user1.png"
 │    ├── Object: "avatar/user2.png"
 │    └── Object: "banner/hero.jpg"
 │
 ├── Bucket: "my-app-logs"
 │    ├── Object: "2024/01/01/access.log"
 │    └── Object: "2024/01/02/access.log"
 │
 └── Bucket: "my-app-backups"
      └── Object: "db/2024-01-01.sql.gz"

注意事项:
  - Bucket 名称全球唯一（DNS 兼容：3-63 字符，小写字母/数字/连字符）
  - Bucket 本身有 Region 属性，数据存储在指定地域
  - 一个 Account 下 Bucket 数量有限（默认 100-1000）
  - Bucket 内 Object 数量无上限
```

### 3.2 CRUSH 算法 —— 数据放置的灵魂

```
Object 寻址过程:

  Object Key: "photos/cat.jpg"
       │
       │ hash(key) → 32位整数
       ▼
  ┌────────────────────────────────────┐
  │ PG 映射:  hash(key) mod pg_num    │
  │ 例: hash("photos/cat.jpg") mod 256│
  │     = PG #47                       │
  └──────────────┬─────────────────────┘
                 │
                 │ CRUSH 算法
                 │ (Controlled Replication Under Scalable Hashing)
                 ▼
  ┌────────────────────────────────────┐
  │ 根据 CRUSH Map 选择 OSD:           │
  │                                    │
  │ CRUSH Rule:                        │
  │   step take root                   │
  │   step chooseleaf firstn 3         │
  │         type host                  │
  │   step emit                        │
  │                                    │
  │ → 选择 3 个不同 host 上的 OSD      │
  │ → OSD.2 (host-a)                   │
  │ → OSD.7 (host-b)                   │
  │ → OSD.15 (host-c)                  │
  └────────────────────────────────────┘

CRUSH 的优势:
  ✅ 去中心化 — 客户端直接计算数据位置，无需中心查找表
  ✅ 拓扑感知 — 副本分布在不同 host/rack/DC
  ✅ 扩展时最小数据迁移 — 新增 OSD 只影响部分 PG
```

### 3.3 写入流程（三副本强一致）

```
Client            Primary OSD        Secondary OSD 1     Secondary OSD 2
  │                    │                    │                    │
  │ ── PUT object ───→│                    │                    │
  │                    │                    │                    │
  │                    │── replicate ──────→│                    │
  │                    │── replicate ──────────────────────────→│
  │                    │                    │                    │
  │                    │                    │── ACK ───────────→│
  │                    │←── ACK ───────────│                    │
  │                    │←──────────────────────── ACK ─────────│
  │                    │                    │                    │
  │ ←── 200 OK ───────│                    │                    │
  │                    │                    │                    │

要点:
  - Primary OSD 接收写入，转发副本到 Secondary
  - 所有副本确认后才返回成功（强一致）
  - PG 内使用 PG Log 做版本追踪和故障恢复
```

## 四、大文件上传 —— Multipart Upload

### 4.1 流程详解

```
┌──────────────────────────────────────────────────────────────────┐
│                 Multipart Upload 流程                              │
│                                                                  │
│  Step 1: Initiate Multipart Upload                               │
│  POST /bucket/large-file.tar.gz?uploads                          │
│  → 返回 UploadId: "abc123"                                       │
│                                                                  │
│  Step 2: Upload Parts (可并发)                                    │
│  ┌────┬────┬────┬────┬────┬────┬────────┐                        │
│  │ P1 │ P2 │ P3 │ P4 │ P5 │ P6 │ P7     │  ← 每个 Part 5MB-5GB │
│  │5MB │5MB │5MB │5MB │5MB │5MB │ 3MB    │  ← 最多 10,000 Parts  │
│  └──┬─┴──┬─┴──┬─┴──┬─┴──┬─┴──┬─┴───┬────┘                       │
│     │    │    │    │    │    │      │                             │
│     ▼    ▼    ▼    ▼    ▼    ▼      ▼                             │
│  PUT /bucket/large-file.tar.gz?partNumber=1&uploadId=abc123      │
│  PUT /bucket/large-file.tar.gz?partNumber=2&uploadId=abc123      │
│  ...                                                             │
│  → 每个 Part 返回 ETag                                           │
│                                                                  │
│  Step 3: Complete Multipart Upload                               │
│  POST /bucket/large-file.tar.gz?uploadId=abc123                  │
│  Body: <Part1 ETag, Part2 ETag, ... Part7 ETag>                  │
│  → 服务端合并所有 Parts 为最终对象                                 │
│                                                                  │
│  容错: 任意 Part 上传失败可重试，不影响其他 Parts                   │
│  清理: Abort Multipart Upload 释放已上传的 Parts                   │
└──────────────────────────────────────────────────────────────────┘
```

### 4.2 关键参数

| 参数 | AWS S3 | GCS | Azure Blob |
|------|--------|-----|------------|
| 单 Part 最小 | 5 MB | 5 MB | N/A (Block) |
| 单 Part 最大 | 5 GB | 5 GB | 4000 MB |
| 最大 Parts 数 | 10,000 | 10,000 | 50,000 |
| 最大对象大小 | 5 TB | 5 TB | 4.75 TB |
| 并发上传 | ✅ | ✅ | ✅ |

## 五、一致性模型

### 5.1 AWS S3 一致性演进

```
 ═══ 2020 年 12 月之前 ═══

  PUT (新对象) → 读后写一致 (Read-after-Write Consistency)
  PUT (覆盖)   → 最终一致 (Eventual Consistency)
  DELETE       → 最终一致

  问题场景:
  Client A: PUT s3://bucket/config.json (v2)
  Client B: GET s3://bucket/config.json → 可能读到 v1  ← 最终一致!

 ═══ 2020 年 12 月之后 (现在) ═══

  所有操作 → 强读后写一致 (Strong Read-after-Write Consistency)

  Client A: PUT s3://bucket/config.json (v2)
  Client B: GET s3://bucket/config.json → 一定读到 v2  ← 强一致 ✓

  ★ S3 通过改进内部元数据索引实现了"free"强一致性
  ★ 无性能损失、无额外成本
```

### 5.2 底层实现原理

```
元数据索引架构 (S3 内部):

  ┌───────────────────────────────────────────────┐
  │  分布式缓存层 (Request Router)                  │
  │  每个 S3 请求首先查询元数据                      │
  └──────────────────┬────────────────────────────┘
                     │
                     ▼
  ┌───────────────────────────────────────────────┐
  │  元数据索引 (B-tree / LSM-tree 变体)            │
  │  存储: Object Key → { version, location,      │
  │         checksum, storage_class, ... }         │
  │                                                │
  │  关键设计:                                      │
  │  - 写操作先原子更新索引，再确认数据               │
  │  - 读操作总是从索引获取最新版本                   │
  │  - 使用 Witness 组件确保跨分区一致性              │
  └───────────────────────────────────────────────┘
```

> **面试关键点**：AWS 在 2020 年实现 S3 强一致性是业界重大突破。之前需要在 S3 前加 DynamoDB 做一致性层，现在完全不需要了。

## 七、数据持久性与可用性 —— 底层原理

### 7.1 11 个 9 的持久性 (99.999999999%)

```
AWS S3 Standard: 99.999999999% 持久性

含义: 存储 10,000,000 (1千万) 个对象
      → 平均每 10,000 年丢失 1 个对象

实现机制:
  ┌──────────────────────────────────────────┐
  │  1. 多 AZ 冗余 (至少 3 个可用区)          │
  │     Object 副本分布在同 Region 的          │
  │     3+ 个物理隔离的数据中心                │
  │                                          │
  │  2. 数据完整性校验                         │
  │     ┌─ 写入时: MD5 校验和验证              │
  │     ├─ 存储时: 周期性后台校验 (Scrubbing)   │
  │     └─ 读取时: ETag 验证                   │
  │                                          │
  │  3. 自动修复                               │
  │     检测到副本损坏/丢失 → 自动从健康副本     │
  │     重建 → 恢复到目标冗余级别               │
  │                                          │
  │  4. 纠删码 (Erasure Coding)               │
  │     低频访问类: 使用 EC 替代三副本           │
  │     例: 10+4 方案 → 14 块中任意 10 块       │
  │     即可恢复数据，空间利用率 71%             │
  └──────────────────────────────────────────┘
```

### 7.2 纠删码 (Erasure Coding) 详解

```
原理 (Reed-Solomon 编码):

  原始数据: 分为 k 个数据块
  编码生成: m 个校验块
  容错能力: 任意丢失 m 个块仍可恢复

  示例: k=10, m=4 (10+4 方案)

  ┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
  │D0│D1│D2│D3│D4│D5│D6│D7│D8│D9│P0│P1│P2│P3│
  └──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
   ↑────── 10 数据块 ──────↑ ↑── 4 校验块 ──↑

  空间效率:  10/14 = 71.4% (三副本仅 33.3%)
  容错能力:  最多丢失 4 个任意块
  代价:     编解码需 CPU 计算（GF(2^8) 矩阵运算）

  三副本 vs 纠删码:
  ┌────────────┬────────────────┬────────────────┐
  │            │  三副本          │  EC (10+4)      │
  ├────────────┼────────────────┼────────────────┤
  │ 空间利用率  │  33.3%         │  71.4%          │
  │ 容错节点数  │  2 个           │  4 个           │
  │ 读延迟     │  低（直接读）    │  低（直接读）    │
  │ 写延迟     │  中（写3份）     │  高（编码计算）  │
  │ 修复带宽   │  低（复制1份）   │  高（需读k块）   │
  │ 适用场景   │  热数据          │  温/冷数据       │
```

### 8.2 Pre-signed URL 原理

```python
# 生成 Pre-signed URL (Python boto3)
import boto3

s3 = boto3.client('s3')
url = s3.generate_presigned_url(
    'get_object',
    Params={'Bucket': 'my-bucket', 'Key': 'secret/report.pdf'},
    ExpiresIn=3600  # 1小时有效
)
# https://my-bucket.s3.amazonaws.com/secret/report.pdf
#   ?X-Amz-Algorithm=AWS4-HMAC-SHA256
#   &X-Amz-Credential=AKIAIOSFODNN7.../20240101/us-east-1/s3/aws4_request
#   &X-Amz-Date=20240101T120000Z
#   &X-Amz-Expires=3600
#   &X-Amz-Signature=fe5f8...3d2a

# 验证流程:
# 1. S3 从 URL 提取签名参数
# 2. 用相同算法 + 请求信息重新计算签名
# 3. 对比签名是否一致
# 4. 检查是否过期
```

## 十一、面试高频问题

### Q1: 对象存储为什么不能"追加写入"或"部分修改"？

**答**：对象存储的数据模型是**不可变的 (Immutable)**。一个对象写入后，只能整体替换（PUT），不能 `seek()` 到中间位置修改。原因：
1. **数据完整性**：对象的 ETag/校验和基于完整内容计算
2. **纠删码**：EC 编码后的数据块是全局相关的，修改一处需要重新编码
3. **缓存与 CDN**：不可变对象可以安全缓存
4. **简化分布式一致性**：无需处理并发修改同一对象的冲突

> Azure Blob 的 Page Blob 是例外，它支持 512B 粒度的随机写入（用于虚拟机磁盘）。

### Q2: S3 的 11 个 9 持久性是怎么实现的？

**答**：
- 多 AZ 冗余存储（至少 3 个物理隔离的数据中心）
- 数据写入时计算 MD5 校验和，存储时周期性后台校验
- 检测到损坏自动从健康副本修复
- 温/冷数据使用纠删码进一步提高空间利用率和容错能力

### Q3: 什么场景选对象存储 vs 块存储 vs 文件存储？

**答**：
- **对象存储**：海量非结构化数据（图片、视频、日志、备份、数据湖）
- **块存储**：需要低延迟随机 I/O 的场景（数据库磁盘、虚拟机卷）
- **文件存储**：需要 POSIX 语义的共享访问场景（多实例共享配置、机器学习训练数据）

### Q4: Pre-signed URL 的安全隐患与防护？

**答**：
- URL 泄露 = 数据泄露 → 设置短有效期（分钟级）
- 限制 IP 范围（Bucket Policy Condition: `aws:SourceIp`）
- 使用 HTTPS 防止中间人窃取 URL
- 审计日志追踪 Pre-signed URL 的生成和使用

### Q5: 如何处理 S3 的 LIST 性能问题？

**答**：S3 LIST 只能返回 1000 个对象/次，百万级文件的 LIST 极慢。
- 用 S3 Inventory 替代实时 LIST（异步生成完整清单）
- 自建元数据索引（DynamoDB / Elasticsearch）
- 合理设计 Key 前缀，用 Prefix + Delimiter 做逻辑分页
