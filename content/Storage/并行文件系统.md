# 并行文件系统硬核面试手册

> 目标：这不是“知道 Lustre/BeeGFS/GPFS 名字就算会”的版本，而是一份能拿去回答 HPC、AI 集群、超算存储、训练数据平台、线上海量吞吐调优问题的高密度手册。  
> 适用场景：GPU 训练集群、EDA、基因组学、渲染农场、超算中心、高带宽共享存储。  
> 阅读建议：优先背 `1. 一句话打穿`、`5. 读写流程`、`7. 上下限边界`、`11. 面试实战点`。

---

## 一、一句话打穿

并行文件系统的核心，不是“大家共享一个目录树”，而是：

> 把一个文件的数据条带化切到多个存储目标上，让多个客户端能并发读写同一命名空间，并用元数据服务、锁、缓存一致性和高速网络，把单机文件系统的语义扩展成集群级高吞吐共享存储。

更面试化一点：

> 并行文件系统 = 全局命名空间 + 分离的元数据/数据路径 + 文件条带化 + 并行 I/O + 锁与缓存一致性 + 故障转移 + 高速互联。

如果面试官只给 30 秒，可以答：

> 并行文件系统的价值，在于把“单文件系统”的共享语义和“分布式存储”的聚合带宽结合起来。它最擅长的是高并发、大文件、顺序吞吐型工作负载，但它的天敌通常是元数据热点、小文件风暴、锁竞争以及故障恢复期间的尾延迟恶化。

## 二、什么是并行文件系统

### 2.1 和普通分布式文件系统的区别

不是所有“多机共享目录”的系统都叫并行文件系统。

并行文件系统通常同时满足几个特征：

1. **单个文件会被切成多个条带并落到多个目标上**。
2. **多个客户端可以同时从多个存储节点并发读写**。
3. **数据通路尽量绕开元数据节点，客户端直连数据节点**。
4. **目标是把总吞吐做大，而不只是把容量做大**。

所以：

- `NFS scale-up NAS` 更像“共享访问”
- `HDFS` 更像“大数据批处理文件层”
- `CephFS / Lustre / BeeGFS / IBM Storage Scale / WEKA` 才更接近严格意义上的并行文件系统语境

### 2.2 为什么它在 HPC / AI 场景特别重要

因为这些场景的典型诉求不是“对象很多”，而是：

- 几百到几千个计算节点同时扫同一批训练集
- 多个作业并发 checkpoint
- 单任务持续拉满数十到数百 GB/s 吞吐
- 需要 POSIX 文件语义，而不是自己改应用去适配对象接口

一句话：

> 对象存储更像“海量数据池”，并行文件系统更像“高性能共享工作盘”。

## 三、核心架构

### 3.1 典型架构图

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                              Compute / Client Layer                        │
│                                                                             │
│  Client-1        Client-2        Client-3        ...        Client-N        │
│  MPI / CUDA      Training Job    Render Worker             Analytics Job    │
│      │               │               │                          │           │
│      └───────────────┴───────────────┴───────────────┬──────────┘           │
│                                                      │                      │
│                          High-Speed Fabric           │                      │
│                   100/200/400GbE or InfiniBand       │                      │
│                                                      ▼                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                           Metadata / Control Plane                          │
│                                                                             │
│   MDS / MDT-1        MDS / MDT-2        Lock Manager / Quota / Namespace    │
│   目录、inode         热目录拆分/HA         条带布局、权限、配额、租约        │
│                                                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                             Data / Storage Plane                             │
│                                                                             │
│   OSS-1            OSS-2            OSS-3            ...        OSS-N       │
│   │                │                │                            │          │
│  OST-1/2          OST-3/4          OST-5/6                       OST-*      │
│  NVMe/HDD         NVMe/HDD         NVMe/HDD                      NVMe/HDD   │
│                                                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                               Capacity Layer                                 │
│                                                                             │
│       RAID / JBOD / EC / Mirror / NVRAM / SSD Tier / HDD Tier / S3 Tier     │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 你必须掌握的组件分工

| 组件 | 作用 | 面试高频点 |
| --- | --- | --- |
| **Client** | 挂载文件系统并发起 POSIX I/O | 页缓存、并发度、挂载参数、RDMA |
| **MDS / Metadata Server** | 管目录、inode、布局、锁、权限 | 元数据热点、小文件瓶颈、Failover |
| **MDT / Metadata Target** | 存元数据对象 | 目录分片、日志恢复 |
| **OSS / Storage Server** | 提供数据对象访问 | 带宽聚合、队列深度、CPU 开销 |
| **OST / Data Target** | 实际存文件条带 | 条带大小、条带数、恢复域 |
| **高速网络** | 承载数据与控制流量 | IB/RDMA、RoCE、拥塞、oversubscription |
| **Lock / Cache Coherency** | 维持并发读写一致性 | 租约、锁粒度、锁回收 |

### 3.3 数据和元数据为什么必须分离

因为如果元数据节点既负责路径解析又负责每个数据包转发，它会变成超级瓶颈。

并行文件系统的经典做法是：

1. 客户端先找元数据节点做 `lookup/open/layout/lock`。
2. 拿到布局信息后，客户端直接并行访问数据节点。
3. 元数据节点留在控制路径，不在大流量数据路径上。

这就是它能扩到很高吞吐的根本原因。

## 四、核心数据模型与关键概念

### 4.1 文件不是完整落在一个盘上

并行文件系统常见的数据布局是：

```text
文件 F = [Stripe-0][Stripe-1][Stripe-2][Stripe-3][Stripe-4]...

Stripe-0 -> OST-1
Stripe-1 -> OST-2
Stripe-2 -> OST-3
Stripe-3 -> OST-4
Stripe-4 -> OST-1
...
```

也就是：

- **条带大小（stripe size）**：每条写多大
- **条带数（stripe count）**：跨多少目标
- **条带模式（layout）**：轮转、固定、分层、按目录策略继承

### 4.2 为什么条带化很重要

它直接决定：

- 单文件最大可用带宽
- 顺序读写的并行度
- 存储目标之间是否均衡
- 恢复和重建时的并发压力

常见误区：

> 条带数不是越大越好。  
> 小文件或中等文件如果盲目打很宽条带，只会放大元数据和锁开销，还会制造读放大。

### 4.3 元数据比你想的更贵

很多人只盯磁盘吞吐，忽略元数据。

实际上这些操作都主要打在元数据面：

- `create`
- `open`
- `close`
- `stat`
- `rename`
- `readdir`
- 权限检查
- 路径解析

因此并行文件系统里有一句很重要的话：

> 大文件问题大多是数据面问题，小文件问题大多是元数据问题。

### 4.4 锁与一致性是性能和语义的交易

并行文件系统通常要支持 POSIX 语义，因此会涉及：

- inode 锁
- extent/range 锁
- 目录锁
- open/close 一致性
- client cache 失效

这意味着：

- 一致性越强，控制开销越大
- 客户端缓存越激进，失效复杂度越高
- 多 writer 共享同一个文件，最容易出现锁争用和抖动

## 五、读写流程

### 5.1 写流程

```text
应用 write()
   │
   ▼
Client 先向 MDS 发起:
   - 路径解析
   - 权限检查
   - open/create
   - 获取 layout（条带信息）
   - 获取锁 / 租约
   │
   ▼
MDS 返回:
   - 文件 inode / handle
   - stripe size / stripe count
   - 目标 OST/Storage Server 列表
   │
   ▼
Client 按 layout 把数据切片:
   - Chunk-0 -> OSS-1/OST-1
   - Chunk-1 -> OSS-2/OST-2
   - Chunk-2 -> OSS-3/OST-3
   ...
   │
   ▼
多个数据目标并发落盘 / 进入服务端缓存 / NVRAM / 日志
   │
   ▼
客户端收集 ACK
   │
   ▼
close/fsync 时回到 MDS:
   - 更新大小、mtime、extent map
   - 提交元数据日志
   - 释放锁
```

### 5.2 写流程要点

面试里可以强调：

1. **元数据节点通常不在大流量数据面上**。
2. **数据写成功不等于元数据最终可见**，还要看 `close/fsync/journal commit`。
3. **checkpoint 风暴**时，MDS、服务端日志、网络突发都会成为问题。
4. **多客户端并发写同一文件**往往比“每人写自己的文件”难很多。

### 5.3 读流程

```text
应用 read()
   │
   ▼
Client 向 MDS:
   - lookup/open
   - 获取 layout
   - 获取必要锁
   │
   ▼
Client 并发向多个存储目标拉取条带
   │
   ├── 读取 Stripe-0 from OST-1
   ├── 读取 Stripe-1 from OST-2
   ├── 读取 Stripe-2 from OST-3
   └── ...
   │
   ▼
Client 重组数据顺序
   │
   ▼
返回给应用 / 进入页缓存
```

### 5.4 读流程的关键优化点

- 预读（readahead）
- 客户端页缓存命中
- 大块顺序 I/O
- 更宽的并发队列
- RDMA zero-copy / kernel bypass 能力
- 目录和文件布局缓存

## 六、为什么它能快，为什么它也会突然很慢

### 6.1 它能快的根本原因

因为吞吐来自“聚合”：

```text
总带宽 ≈ min(
  客户端总出流能力,
  网络双向截面带宽,
  所有存储节点网卡总带宽,
  所有存储设备可持续吞吐,
  元数据平面能支撑的控制请求速率
)
```

举例：

- 32 个客户端
- 每个客户端 12 GB/s
- 8 个存储节点
- 每节点 50 GB/s 可持续读

理想上限不是 `32 * 12 = 384 GB/s`，而是还要看：

- fabric 截面是否够
- 服务端是否能吐满 `8 * 50 = 400 GB/s`
- 元数据是否拖后腿

### 6.2 它会慢的典型原因

| 现象 | 根因 |
| --- | --- |
| 大文件吞吐上不去 | 条带太窄、服务端不均衡、网络拥塞 |
| 小文件大量创建慢 | MDS 打满、目录锁冲突、journal 压力 |
| 多作业一起抖动 | noisy neighbor、服务端 CPU 打满、NIC 队列争用 |
| 单个热点目录异常慢 | 目录项过多、单 MDT 热点 |
| checkpoint 时全系统发抖 | 突发写、刷盘回压、元数据提交风暴 |
| 故障后性能大跌 | failover、重建、degraded mode |

## 七、上下限边界

这一节是面试拉开差距的重点。

### 7.1 理论上限边界

#### 带宽上限

```text
单文件顺序读写上限
≈ min(
  stripe_count × 单目标吞吐,
  客户端 NIC,
  客户端 CPU / 内存拷贝能力,
  fabric 到这些目标的有效带宽
)
```

#### 小文件上限

小文件 workload 的上限，通常不是磁盘，而是：

```text
小文件吞吐上限
≈ min(
  MDS ops/s,
  open/close/lookup RTT,
  目录锁竞争能力,
  journal commit 速率
)
```

这也是为什么：

> 一个能跑 200 GB/s 大文件顺序吞吐的并行文件系统，完全可能在“小文件每秒 create 数”上表现得很一般。

### 7.2 下限边界

并行文件系统的“下限”通常出现在这些时刻：

1. **单 MDS 热目录**  
   一个目录挂几千万文件，所有训练进程一起 `readdir/stat/open`

2. **全员 checkpoint**  
   数百个作业同时刷模型和优化器状态

3. **Failover / Rebuild**  
   一个存储节点下线后，其它节点要额外承担恢复流量

4. **弱网络拓扑**  
   Leaf-Spine oversubscription 太高，东西向拥塞严重

5. **错误的条带策略**  
   小文件条带太宽，大文件条带太窄

### 7.3 实战中真正的边界

真正限制系统的，经常不是“盘不够快”，而是：

- MDS 延迟
- 目录结构设计
- 客户端并发模式
- 网卡和 NUMA 绑定
- 机架级网络截面
- 作业调度是否懂存储 locality
- 恢复窗口是否把线上吞吐吃掉

一句话总结：

> 并行文件系统的上限主要由数据面决定，下限主要由控制面和异常场景决定。

## 八、对硬件的要求

### 8.1 网络要求往往比磁盘更先到瓶颈

并行文件系统几乎都依赖强网络：

- **100/200/400GbE** 或 **HDR/NDR InfiniBand**
- 尽量支持 **RDMA**
- 低 oversubscription 的 Clos / Spine-Leaf
- 足够的 PCIe lane 和 NUMA 亲和性

如果网络不行，再快的后端盘也会被闷死。

### 8.2 元数据节点硬件要求

元数据节点看重的是：

- **低延迟**
- **高 IOPS**
- **稳定的 fsync/journal latency**

典型要求：

- 高主频 CPU
- 低延迟 NVMe
- 电容保护写缓存 / NVRAM
- 双机 HA
- 充足内存做目录/inode cache

元数据节点不是“容量机器”，而是“延迟机器”。

### 8.3 存储节点硬件要求

这部分更像“带宽机器”：

- 大量 NVMe 或者 HDD + SSD cache/tier
- 多队列高带宽网卡
- 足够 CPU 做协议、校验、EC、压缩
- 高 PCIe 带宽
- 机箱散热和持续写稳定性

### 8.4 客户端要求

别忽略客户端。

客户端如果配置差，也会让系统看起来像存储坏了：

- 挂载参数不合理
- 页缓存太小
- NUMA 跨 socket
- IRQ 没绑核
- I/O 深度不够
- 应用层并发度太低

### 8.5 GPU / AI 场景的附加要求

AI 场景往往要求：

- GPU 集群与存储网络同等级别设计
- 训练节点具备高速 NIC
- 数据集读取采用 shard/pack 方式降低小文件放大
- checkpoint 需要写放大可控
- 多租户 GPU 作业对存储 QoS 有基本隔离

## 九、业界厂商与产品特点

这一节不要背成品牌列表，要背“架构风格差异”。

### 9.1 Lustre

关键词：

- 经典 HPC 并行文件系统
- MDS/MDT 与 OSS/OST 强分离
- 大文件顺序吞吐非常强
- 在超算、科研、气象、EDA 场景历史包袱和生态都很重

特点：

- 架构清晰，行业认知度高
- 对大带宽工作负载非常成熟
- 条带、MDT/OST、恢复域等概念非常典型，适合面试答题
- 小文件、元数据热点、运维复杂度通常需要更强经验来兜

适合回答成：

> Lustre 是最经典的“高吞吐优先”并行文件系统代表，特别适合解释元数据/数据平面分离、条带化以及为什么单文件能横跨多个目标并发读写。

### 9.2 BeeGFS

关键词：

- 部署灵活
- 元数据和存储服务拆分清楚
- 在 AI / HPC 集群里常见
- 运维体验通常比传统超算体系更轻一些

特点：

- 易于扩容与落地
- 支持按目录继承/控制条带
- 元数据水平扩展思路清晰
- 常被拿来做“相对好上手”的并行文件系统方案

### 9.3 IBM Storage Scale（原 GPFS）

关键词：

- 企业级 + HPC 双栖
- 功能面非常强
- 多协议、多站点、策略引擎、自动分层、全局命名空间都比较成熟

特点：

- 不只是高性能文件系统，更像完整的数据管理平台
- 对混合工作负载、企业特性、跨站点访问、策略编排支持更强
- 面试里常被用来考“为什么企业场景不只看大带宽”

### 9.4 WEKA

关键词：

- 软件定义
- 强调现代 CPU/NVMe/RDMA/AI 训练吞吐
- 对全闪、高并发、低延迟和 GPU 数据喂给更友好

特点：

- 更偏“现代数据平台 + 并行文件系统”
- 对云上、容器化、AI pipeline 集成叙事很强
- 常见卖点是极高吞吐与低延迟并存，以及简化运维

### 9.5 VAST Data

关键词：

- 面向 AI 数据平台
- 偏全闪、解耦、全局命名空间、多协议
- 更接近“现代海量共享存储平台”，不完全是传统并行文件系统形态

特点：

- 强调大规模共享数据、训练数据供给、统一命名空间
- 文件、对象、分析型访问融合趋势明显

### 9.6 DAOS

关键词：

- HPC / Exascale 背景
- 更偏对象化底座
- POSIX 通常通过上层接口提供

特点：

- 很适合回答“新一代高性能存储为什么不一定从 POSIX 语义出发”
- 适合对比传统并行文件系统与对象原生架构的差异

### 9.7 云上托管形态

云上常见是“托管 Lustre / 高性能共享文件服务”：

- Amazon FSx for Lustre
- Azure Managed Lustre
- 各家高性能 NAS / 并行共享文件服务

云上产品特点：

- 部署快
- 扩容简单
- 运维负担低
- 但你对底层布局和硬件控制权更弱，成本模型也更要小心

### 9.8 厂商差异怎么答

不要答成“这个好，那个也好”。

建议按这 5 个维度比较：

1. 元数据扩展方式
2. 数据面架构是传统分层还是更现代的解耦/全闪
3. 小文件和混合负载能力
4. 多协议、跨站点、分层与对象打通能力
5. 运维复杂度与硬件依赖

## 十、和分布式对象存储的区别

这是高频必考题。

### 10.1 一张表讲清楚

| 维度 | 并行文件系统 | 分布式对象存储 |
| --- | --- | --- |
| 抽象 | 文件 / 目录 / POSIX namespace | Bucket / Object / Metadata |
| 访问接口 | POSIX、挂载、文件语义 | HTTP REST / SDK / S3 API |
| 典型优势 | 共享文件语义 + 高吞吐并发 I/O | 海量容量、低成本、多租户、弹性强 |
| 元数据压力 | 路径、inode、目录操作很重 | 对象索引与 bucket 元数据更重要 |
| 单文件/单对象更新 | 支持文件偏移读写 | 通常以整对象 PUT/GET 为主 |
| 多客户端共享修改 | 依赖锁与一致性协议 | 通常不走 POSIX 共享写 |
| 最适场景 | 训练数据、EDA、超算、渲染、共享工作盘 | 图片、备份、归档、数据湖、分发 |
| 典型瓶颈 | 元数据热点、锁竞争、小文件风暴 | 小对象风暴、列举/list、跨区一致性、延迟 |

### 10.2 从应用视角看

如果应用是：

- `open/read/write/fsync/rename`
- 依赖目录结构
- 依赖 POSIX 语义
- 需要共享文件工作区

那通常更偏并行文件系统。

如果应用是：

- PUT/GET
- 大量静态对象
- 归档、备份、数据湖
- 跨区域、多租户、成本优先

那通常更偏对象存储。

### 10.3 从系统设计视角看

并行文件系统最难的是：

- **强语义**
- **共享写**
- **元数据**
- **锁**

对象存储最难的是：

- **超大规模命名空间**
- **跨可用区/跨地域复制**
- **低成本**
- **海量多租户隔离**

一句话：

> 并行文件系统解决的是“很多机器像访问一块共享高性能工作盘”，对象存储解决的是“海量对象如何在成本、可靠性、弹性之间做全局最优”。

## 十一、发展趋势

### 11.1 全闪、NVMe、RDMA、GPU Direct 化

传统并行文件系统更多建立在 HDD 阵列时代。

现在的趋势是：

- 全闪化
- NVMe over Fabrics
- RDMA
- 更靠近 GPU pipeline
- 更强调端到端低延迟和高并发小块 I/O

### 11.2 文件与对象融合

越来越多平台在做：

- 热数据走并行文件系统
- 冷数据或长期留存落 S3 / 对象存储
- 文件层和对象层通过 HSM / tiering / namespace bridge 打通

也就是：

> 文件系统不再是孤立产品，而是“高性能热层 + 对象冷层”的一部分。

### 11.3 小文件优化越来越重要

AI 和数据科学把“小文件问题”放大了：

- 样本图像
- tokenized shards
- 特征文件
- checkpoint 元信息

所以现在系统越来越重视：

- metadata scale-out
- 目录分片
- 小文件打包
- namespace cache
- client-side aggregation

### 11.4 从“超算专用”走向“AI 数据平台”

过去并行文件系统是典型 HPC 词汇。

现在它正越来越多出现在：

- 大模型训练
- 多模态数据准备
- 向量与特征处理
- 实时/近实时数据供给

因此产品路线也从“纯 HPC 带宽”转向：

- 易运维
- 云集成
- Kubernetes/容器友好
- 多协议
- 文件 + 对象 + 数据管理一体化

## 十二、设计与选型时必须问的硬问题

### 12.1 你的主负载是什么

先问这几个：

1. 大文件顺序读写，还是小文件随机访问？
2. 读多写少，还是 checkpoint 型突发写？
3. 是否依赖 POSIX 完整语义？
4. 是否需要多租户隔离？
5. 是否必须跨站点？
6. 热数据保留多久？冷数据要不要落对象层？

### 12.2 条带怎么选

经验上：

- 大文件顺序流：更宽条带、更大 stripe size
- 小文件密集：更窄条带，甚至避免无意义条带扩散
- 混合负载：按目录或工作负载类型做策略隔离

错误案例：

> 100KB 小文件也打到 16 个 OST 上，结果每次 open/read 都放大为跨多个目标访问，系统被自己的“并行”拖慢。

### 12.3 什么时候别选并行文件系统

如果你的场景是：

- 冷归档
- 图片/视频静态分发
- 海量低频对象
- 跨区域海量复制
- 成本绝对优先

那对象存储通常更合适。

## 十三、面试会考的实战点

这一节建议直接背。

### 13.1 设计题

#### 题 1：给 512 张 GPU 的训练集群设计共享存储

你要主动回答：

- 单节点期望吞吐多少
- 聚合吞吐目标多少
- 数据集总量、热点比例、checkpoint 规模
- 元数据操作模型
- 客户端网络与存储网络是否分离
- 为什么用并行文件系统而不是纯对象存储
- 冷数据是否分层到对象层

#### 题 2：为什么训练开始时特别慢

典型根因：

- 小文件风暴
- 所有 worker 同时扫描目录
- readdir/stat/open 打爆 MDS
- DataLoader 并发模型不合理

优化思路：

- 把小文件打包成 shard
- 预生成 manifest
- 减少目录遍历
- 做热点预热和元数据缓存

#### 题 3：为什么 checkpoint 时全盘抖动

典型原因：

- 突发写太大
- 所有任务同时 fsync
- journal / service log / NVRAM 刷写受限
- 服务端网络与数据盘同时承压

优化：

- 错峰 checkpoint
- 增大写入并发但控制节奏
- 分层缓存
- 调整每作业写入策略和 chunk 大小

### 13.2 排障题

#### 题 4：带宽只有理论值的 1/5

排障顺序：

1. 看客户端并发度够不够
2. 看单客户端 NIC 是否打满
3. 看 fabric 是否拥塞
4. 看 OSS/存储节点 CPU / NIC / queue depth
5. 看条带是否过窄
6. 看是否被单一热点目录或锁限制

#### 题 5：单目录几千万文件怎么办

高频答法：

- 目录分层
- 哈希打散
- 多 MDT / metadata sharding
- 避免运行时全量 list
- 用 manifest / index 替代目录扫描

#### 题 6：一个存储节点挂了会怎样

正确答法应覆盖：

- 数据可用性是否靠 RAID / mirror / EC / buddy mirror / failover
- 控制面是否需要接管
- degraded mode 下吞吐为什么下降
- 恢复窗口内为什么线上业务也会抖

### 13.3 面试官最爱追问的点

1. 为什么并行文件系统更怕小文件？
2. 为什么对象存储不适合直接替代训练工作盘？
3. 元数据节点和数据节点为什么要分离？
4. 条带宽度为什么不能一味做大？
5. 为什么网络经常先于磁盘成为瓶颈？
6. 多客户端同时写同一文件会遇到什么问题？
7. failover/rebuild 为什么会伤害尾延迟？

## 十四、可以直接背的高质量回答

### 14.1 什么是并行文件系统

> 并行文件系统是面向高吞吐共享存储场景的一类分布式文件系统。它通过把文件条带化到多个存储目标、将元数据路径和数据路径分离、让客户端直接并行访问多个数据节点，从而把单机文件系统的共享语义扩展为集群级高带宽访问能力。

### 14.2 它和对象存储最大的区别

> 最大区别不是“一个是文件、一个是对象”这么简单，而是语义和目标不同。并行文件系统追求的是 POSIX 共享语义下的高并发 I/O，对多客户端共享工作区特别重要；对象存储追求的是海量容量、低成本、多租户和广域弹性，更适合作为数据湖和归档层。

### 14.3 它最怕什么

> 最怕三件事：元数据热点、小文件风暴和错误的并发模型。很多系统看起来是磁盘瓶颈，实际上是 MDS、目录锁、网络截面或者客户端访问模式出了问题。

### 14.4 什么时候该选它

> 当业务强依赖 POSIX 文件语义、需要很多计算节点共享同一数据集、对顺序带宽和聚合吞吐要求很高时，优先考虑并行文件系统。典型就是 HPC、EDA、渲染和大模型训练工作盘。

## 十六、最后一句总结

> 如果对象存储解决的是“海量数据怎么便宜、可靠、弹性地存”，那并行文件系统解决的就是“很多机器怎么像共享一块超高速工作盘一样同时干活”。前者偏容量与平台，后者偏吞吐与工作区；前者难在规模和成本，后者难在语义、锁、元数据和尾延迟。
